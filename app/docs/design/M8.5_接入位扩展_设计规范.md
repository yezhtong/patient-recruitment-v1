## 〇、核心决策摘要（给工程师 2 分钟读懂）

| 维度 | 决策 | 一句话理由 |
| --- | --- | --- |
| **社区分区头图** | **复用 M8.4 `community_cover` 规格**（4:3、`var(--r-xl)`、overlay 可选） | 已落地 §2.1，无需重复造；本文件只规定**逐分区 key 接入路径** |
| **试验详情顶图** | **新增"标题正上方"位置**（不挤掉现有 `.trial-hero__top`），16:9、`var(--r-lg)`、**overlay 强制关** | 详情页已有 tag+badge+title，再叠 overlay 会变脏，与 M8.4 §2.2 决策一致 |
| **FAQ 配图** | **hero 模式 21:9 扁条**，**逐条配图 PM 一票否决**（理由见 §三末） | 逐条配图会让 23 条 FAQ 视觉噪音爆炸，且大部分问题（费用/隐私）无对应实拍素材 |
| **欢迎 AI 消息** | **走"评论流里多一条"路线**，**不做 toast** | 已有 `triggerWelcomeIfFirstVisit` 写 DB；toast 会在用户离开页面后丢失上下文，违背"AI 留痕可追溯"原则 |
| **欢迎消息出现动效** | **首次进分区时该条评论 `slide-up + fade-in 360ms`**，仅触发一次 | 让用户感知"这是为我刚生成的"，又不喧宾夺主 |
| **注册推荐分区弹窗** | **弹窗 = 全屏 modal（移动端）/ 居中卡片（桌面）**，**不是 toast 不是 banner** | 注册流末尾的"启动盘"，必须强可见；中老年用户对底部 toast 几乎无感 |
| **弹窗关闭策略** | **必须显式按 ×、"稍后再说"、或主 CTA**；**禁止点击遮罩关闭**、**禁止 ESC 关闭** | 中老年用户对外侧空白点击毫无防备，常常误关；ESC 也少有人用 |
| **弹窗推荐数量** | **固定 3 个分区**（≥ 3 时取前 3，< 3 时按实际数渲染，0 个则跳过弹窗） | 3 是"选择心理学"甜区，超出会让用户动用决策力 |
| **token 新增** | **0**（全复用 design-system.css + M8.4 已建 class） | |
| **新增局部 class** | `.trial-cover`、`.faq-hero`、`.welcome-msg--enter`、`.recommend-modal*` 共 4 组 | 只在 4 个对应 styles.css 内新增，不污染全局 |

## 一、社区分区头图视觉规格（接入位 4）

> M8.4 §2.1 已规定通用 `community_cover` 视觉。本节只规定**接入到 `app/src/app/community/[slug]/page.tsx` 的具体位置 + 逐 slug 数据消费方式**。

### 1.1 容器位置

挂在 `.cm-shell > .container` 内，**`.cm-breadcrumb` 之后**、**`.cm-hero` 之前**。

```
container
├── .cm-breadcrumb           ← 已有
├── .cm-cover                ← 新增 ← 本节
├── .cm-hero                 ← 已有
├── .cm-disclaimer           ← 已有
└── ...
```

### 1.2 容器规格（复用 M8.4 §2.1）

| 属性 | 值 |
| --- | --- |
| class 命名 | `.cm-cover`（外壳）+ `.photo-frame.photo-frame--wide`（内层） |
| 比例 | `aspect-ratio: 4 / 3` |
| 圆角 | `var(--r-xl)` = 20px |
| 阴影 | 无（已有内层 `.photo-frame` border） |
| margin | `margin: 0 0 24px;`（与下方 `.cm-hero` 拉开）|
| overlay | **可选**（每条素材独立开关），开启时 label+text 走 §1.4 |

### 1.3 断点尺寸

| 断点 | 容器最大高度 | 图比例 | overlay padding |
| --- | --- | --- | --- |
| ≥ 1280 | 440px | 4:3 | `28px 32px 24px` |
| 960 – 1279 | 380px | 4:3 | `24px 28px 22px` |
| 641 – 959 | 280px | 4:3 → **3:2 降高** | `20px 22px 18px` |
| 401 – 640 | 200px | **16:9** 横构图（手机宽屏更自然，避免左右黑边） | `16px 18px 14px` |
| ≤ 400 | 180px | 16:9 | `14px 14px 12px` |

> 注：≤ 640 由 4:3 切到 16:9 是为了消除中老年用户在窄屏看到"图被裁很厉害"的感觉；M8.4 §1.1 hero 也用了同类降高思路。

### 1.4 overlay 规格（复用 M8.4 §1.2-1.3）

- 位置：贴**左下**（`left:0 bottom:0 right:auto`，最大宽 `min(60%, 380px)`），与 hero 的"贴底通栏"区分开，更像编辑感封面
- 内边距：见 §1.3 表
- 背景：`linear-gradient(135deg, rgba(10,20,17,.78) 0%, rgba(10,20,17,.45) 60%, transparent 100%)`（135° 取代 hero 的 90°，让右上有亮区放素材主体）
- label 字号 11px / `var(--lime)` / `var(--font-mono)` `.12em` 字距
- text 字号 桌面 22px / ≤640 18px / ≤400 16px / `var(--font-serif)` / `var(--cream-50)`
- 对比度：≥ 15:1（沿用 M8.4 §1.3 计算）

### 1.5 数据消费

- key 形式：`community_cover` + `cover_key = group.slug`（如 `cover_key=lung-cancer`、`breast-cancer`）
- 后台单分区可上传多张，前台 server component 取 **`order ASC` 第 1 条 `isEnabled=true`** 的素材；不轮播
- 命中 0 条 → 走 §1.6 降级

### 1.6 无素材降级

```css
.cm-cover--placeholder {
  background: linear-gradient(135deg, var(--ink-100) 0%, var(--cream-200) 100%);
  display: grid;
  place-items: center;
  color: var(--gray-500);
  font-family: var(--font-serif);
  font-size: clamp(32px, 5vw, 48px);
  letter-spacing: -0.01em;
}
```

中心显示分区名（来自 `group.name`），不出现"暂无图片"等错误感文案。

### 1.7 a11y

```html
<figure class="cm-cover" role="img" aria-label="肺癌 病友社区头图">
  <div class="photo-frame photo-frame--wide">
    <img src="..." alt="" loading="lazy" />
    <div class="cm-cover__overlay">
      <span class="cm-cover__label">★ 病友互助</span>
      <p class="cm-cover__text">和走过同样路的人一起聊。</p>
    </div>
  </div>
</figure>
```

- `<img alt="">` 留空（装饰性），由外层 `<figure aria-label>` 提供语义
- overlay 内的 label/text 是真实内容，不需要 aria-hidden

## 三、FAQ 配图视觉规格（接入位 6）

### 3.1 设计取舍（需 PM 一起讨论 · 见末尾"风险"）

PRD 阶段尚未拍板"hero 还是逐条"。本设计规范**强烈推荐 hero 模式**，三个理由：

1. **23 条 FAQ 逐条配图视觉噪音爆炸**——FAQ 已经是 details 折叠，每条配图等于打开后再"塞"一张图，破坏"读问答"的纯文本节奏
2. **"费用与医保"、"隐私保护"这类问题没有对应的真实素材**——只能用插画或 stock photo，违反我们"不用卡通插画"的设计原则
3. **hero 一张图就足以承载"心安"的情绪**——FAQ 页的核心情绪诉求是"被理解"，一张温和的真实场景图（医生与患者交谈）即可建立

下文规格按 hero 模式给出。若 PM 拍板逐条配图，本节 §3.2-§3.5 作废，需另起小节。

### 3.2 容器位置

挂在 `app/src/app/faq/page.tsx` 内，**eyebrow + h1 + lead 段落组合之后**、**双列 grid（左侧分类导航 / 右侧主内容）之前**：

```
container
├── div (eyebrow + h1 + lead)   ← 已有，行 73-91
├── .faq-hero                   ← 新增 ← 本节（margin-top: -8px 弥补 lead 自带的间距）
├── grid (240px 1fr)            ← 已有，行 93-213
└── ...
```

### 3.3 容器规格

| 属性 | 值 |
| --- | --- |
| class 命名 | `.faq-hero` + `.photo-frame.photo-frame--wide`（覆写 aspect 为 21:9） |
| 比例 | `aspect-ratio: 21 / 9`（扁条，不抢主内容） |
| 圆角 | `var(--r-xl)` = 20px |
| margin | `margin: 8px 0 40px;` |
| overlay | **可选**（默认关；若开启只放一句安抚语，居中，最大 28ch） |

### 3.4 断点尺寸

| 断点 | 容器最大高度 | 图比例 | overlay 字号 |
| --- | --- | --- | --- |
| ≥ 1280 | 280px | 21:9 | 24px serif |
| 960 – 1279 | 240px | 21:9 | 22px serif |
| 641 – 959 | 200px | 21:9 → **3:1 略降高** | 20px serif |
| 401 – 640 | 140px | **2:1**（21:9 在窄屏只剩 ~155px，过扁） | 17px serif |
| ≤ 400 | 120px | 2:1 | 16px serif |

### 3.5 overlay 规格（开启时）

- 位置：**居中**（不同于社区的左下、hero 的贴底）
- 文案：单行优先，溢出自动 2 行
- 颜色：`var(--cream-50)`
- 背景：**全屏柔化遮罩** `linear-gradient(180deg, rgba(10,20,17,.40), rgba(10,20,17,.55))`（不带亮渐变，全图压暗 40-55%）

> 居中遮罩适合"安抚语"语气，左下/贴底更适合"导引"语气。

### 3.6 数据消费

- key 形式：`faq_hero`（无子 key，只取 1 张）
- 命中 0 条 → **不渲染整个 `.faq-hero` section**（不留灰框，避免"残缺感"）

### 3.7 a11y


## 四、注册推荐分区弹窗规格（接入位扩展 · 重点）

### 4.1 触发时机

注册流第 3 步（`/auth/register/records`）**点"完成并进入我的主页"** → server action 完成后 → **重定向到 `/me?recommend=communities`** → `/me` server component 检测到该 query 参数 → 渲染弹窗（client component）。

> 不在 `/auth/register/records` 页内弹——该页已经是 server component + 上传表单，弹窗加进去会让"完成"按钮的点击反馈延迟。

### 4.2 弹窗形态

| 属性 | 值 |
| --- | --- |
| class 命名 | `.recommend-modal`（遮罩）+ `.recommend-modal__card`（卡片） |
| 实现方式 | **原生 `<dialog>` 元素 + `dialog.showModal()`**（标准 a11y、自带焦点陷阱、自带遮罩） |
| 桌面尺寸 | `max-width: 560px`，`width: calc(100vw - 48px)` |
| 移动尺寸 | **全屏**（`width:100vw; height:100vh; max-width:none; border-radius:0`）|
| 圆角 | 桌面 `var(--r-2xl)` = 28px；移动 0 |
| 阴影 | 桌面 `var(--shadow-lg)`；移动无 |
| 背景 | `var(--cream-0)` |
| 遮罩 | `dialog::backdrop { background: rgba(10,20,17,.55); backdrop-filter: blur(4px); }` |
| z-index | 自然走 `<dialog>` top layer，**不需要写 z-index** |

### 4.5 推荐 0 / 1 / 2 个时的降级

| 推荐数 | 行为 |
| --- | --- |
| 0 | **不弹窗**（server 端检查 `recommendations.length === 0` 直接跳过 `?recommend=communities` 写入） |
| 1 | 单卡渲染，标题改为「为你推荐这个分区」 |
| 2 | 双卡渲染，标题改为「这 2 个分区适合你」 |
| ≥ 3 | 取前 3 个，标题保持「这 3 个分区适合你」 |

### 4.6 关闭策略（中老年友好底线）

| 关闭方式 | 是否启用 | 理由 |
| --- | --- | --- |
| 点击右上 × | ✓ 启用 | 主关闭通道，44×44 触摸区（按钮本体 32×32 + padding 6px） |
| 点击「稍后再说」 | ✓ 启用 | 明确语义按钮 |
| 点击「全部加入并去看看」 | ✓ 启用 | 完成意图后自动关 |
| 点击单个「加入」 | ✗ 不关闭 | 保留弹窗让用户继续选其他分区 |
| 点击外部遮罩 | **✗ 禁用** | 中老年用户对模态遮罩无感，常误关；通过 `<dialog>` 配合 `onClick={(e)=>e.target===dialogRef.current && e.preventDefault()}` 阻断 |
| 按 ESC 键 | **✗ 禁用** | 同上；通过 `onCancel={(e)=>e.preventDefault()}` 阻断 dialog 默认行为 |

### 4.7 出现/消失动效

```css
.recommend-modal[open] {
  animation: rec-modal-enter 280ms var(--ease-out);
}
.recommend-modal[open]::backdrop {
  animation: rec-backdrop-enter 280ms var(--ease-out);
}

@keyframes rec-modal-enter {
  from { opacity: 0; transform: translateY(16px) scale(.98); }
  to   { opacity: 1; transform: translateY(0) scale(1); }
}
@keyframes rec-backdrop-enter {
  from { opacity: 0; }
  to   { opacity: 1; }
}

@media (prefers-reduced-motion: reduce) {
  .recommend-modal[open],
  .recommend-modal[open]::backdrop { animation: none; }
}
```

- **不做退场动画**——中老年用户期望"按下立刻关掉"，退场反而让人误以为没关上
- 时长 280ms（M8.4 已锁定 ≤ 480ms 上限）

```css
  }

```

## 五、欢迎 AI 消息规格

### 5.1 形态判定

**走 §一·〇 决策：评论流里多一条**，**不做 toast**。

具体落点：用户**首次**进入某社区分区下任一帖子的详情页（`/community/posts/[id]`），server component 检测「该用户在该 group 没看过任何帖」→ 触发 `triggerWelcomeIfFirstVisit({ userId, groupId })` → 写入一条 `authorRole: "ai"` 的 `comment`，关联到该 post → 评论列表自然渲染该条。

> 这条评论是真实落库的，下次再访问也会看到（不是临时 UI），符合 M8.3 T11 设定。

### 5.2 评论卡视觉规格（沿用 `.cm-comment` 已有样式 + 微差异）

| 属性 | 值 |
| --- | --- |
| 基础 class | `.cm-comment` 已有（community.css §cm-comment 行 514-535） |
| AI 标识 class | `.cm-comment--ai`（**本次新增**） |
| 背景 | `linear-gradient(135deg, var(--lime-soft) 0%, var(--cream-0) 100%)` |
| 边框 | `1px solid var(--lime)` |
| 左边框强调 | `border-left: 4px solid var(--lime)` |
| RoleBadge | 复用 `<RoleBadge role="ai" size="sm" />`（已有 design-system.css 行 1382 ai 样式） |
| 头像 | 复用 `.cm-avatar.cm-avatar--official`（lime 底 ink-900 字，已有 community.css 行 466-468） |
| 内容字号 | 15px（沿用 `.cm-comment .content`） |

### 5.3 入场动效

仅在「这条评论是本次会话**新生成**」时触发（server 通过 query param `?welcome=1` 或 cookie 标记，client 读到后给该条加 `.welcome-msg--enter` 一次）：

```css
.welcome-msg--enter {
  animation: welcome-msg-enter 480ms var(--ease-out);
}

@keyframes welcome-msg-enter {
  0%   { opacity: 0; transform: translateY(12px); }
  60%  { opacity: 1; transform: translateY(0); }
  100% { opacity: 1; transform: translateY(0); }
}

@media (prefers-reduced-motion: reduce) {
  .welcome-msg--enter { animation: none; }
}
```

- 时长 480ms（与 hero crossfade 同节奏）
- 触发后 1.5s（动效完毕后）js 移除该 class，避免下次 hover/repaint 重放
- **不带闪光、不带 bounce**——AI 消息是"陪伴"语气，不是"成就解锁"

### 5.4 出现位置

- 评论列表内**始终置顶**（按 `createdAt` 排序时 AI 用户的评论会因为 createdAt 早而自然在顶部，无需特殊排序）
- 不悬浮、不固定，跟随帖子滚动
- 移动端无差异

### 5.5 a11y

- `.cm-comment--ai` 的 `<RoleBadge>` 已带 `title=` tooltip 解释"AI 助理"含义
- 入场动画 **不影响读屏器**——读屏器读 DOM 文本，不读 transform
- 用户被动看到，无需 focus management
