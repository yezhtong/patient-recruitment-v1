# M6.5 / V2 延期清单

> 来源：M6 第二阶段产品 review 的"建议补"与"必须补但本轮不修"。tech-lead 与 PM 于 2026-04-20 分流后归档于此。
> 规则：M6.5 = M6 收尾后 / M7（部署）前插入的小迭代；V2 = 平台第二大版本。
> 每条必须写清「为什么延期」，否则后续 owner 会误判是新需求。

---

## M6.5 档（M6 收尾后 / M7 前做，0.5–1 天可完成）

### MR-1 · 推广来源维度看板卡
- **出处**：PM 补充点 T1·3
- **内容**：工作台加一张卡「近 30 日线索来源 Top 5」，按 `Lead.sourcePage` groupBy，列「来源 / 线索数 / 占比」
- **为什么延期**：当前 sourcePage 只有 `/trials/[slug]` 和 `/prescreen/[slug]` 两种落地页值，没有渠道参数区分（utm_source 等）。现在做出来只会显示 2 条空洞信息。需等 UTM 渠道追踪在 M6.5 一起做
- **前置依赖**：UTM 参数采集（发帖链接带 `?utm_source=xxx&utm_medium=xxx`，预筛入库时解析存入 sourcePage 扩展字段或新字段）
- **对外口径**：若老板问「渠道 ROI 什么时候看」，回答「UTM 渠道追踪在 M6.5 集成，预计 0.5 天」

### MR-2 · 敏感词第一版种子词库由 team-lead 审定
- **出处**：PM 补充点 T5·4 + PM 专属护栏
- **内容**：现 `data/sensitive-words.txt` 的 35 条词是 dev 默认库，上线前必须由 team-lead（最终负责人）人工过目审定，confirmed 后再批量导入 DB
- **为什么延期**：V1 功能层允许运营自己管理没问题，但**第一版基线词库**涉及法律合规与患者体验判断（如「免费体验」这种词是否允许），超出运营个人授权范围
- **动作人**：PM 直接向 team-lead 提，不派给工程
- **验证**：审定后的词库跑一轮 `bulkImportSensitiveWords`，导入结果写 `DEVELOPMENT_LOG.md`

## V2 档（M7 部署后的第二大版本，单独立项）

### V2-1 · Top 5 试验表加「线索→报名」转化率列
- **出处**：PM 补充点 T1·2
- **为什么延期**：现 `getTopTrials` 用 `Promise.all` 并发批量查询，加一列转化率需要再加一轮 `application.count({ where: { trialId, createdAt: {gte} }})` 的并发。单独改无所谓，但 ROI 归因要和 UTM 渠道一起设计才有业务意义，否则两边口径打架
- **拟合入口**：与 MR-1 同批（M6.5 做完 UTM）再上，或延后到 V2 完整改造看板

### V2-2 · 时间切换补「今日」档
- **出处**：PM 补充点 T1·4
- **为什么延期**：运营日会要看"今天到现在"的数量，但 V1 以「7 天 / 30 天」趋势观察为主，今日数据波动大不适合做决策。V2 加

### V2-3 · KPI 卡同比小三角
- **出处**：PM 建议点 T1·1
- **内容**：每张主指标卡加「vs 上周 / 上月 ↑12%」绿红三角
- **为什么延期**：已有 `DeltaBadge` 组件支持，但需要决定「哪些指标算反向」（pending-* 上升是坏事），一并加 `inverse` 标记。V2 一起做看板视觉升级

### V2-4 · 导出按钮旁显示「上次导出时间」
- **出处**：PM 建议点 T2·2
- **为什么延期**：需要 `export` 审计日志按 operator 维度查最近一条。MVP 可不做

### V2-5 · Application 导出加「项目答案摘要」列
- **出处**：PM 补充点 T2·3
- **为什么延期**：`projectAnswers` 是 JSON 字段，拼接格式需要好好设计（问题顺序、长答案截断、特殊字符转义），不是一个 map 就能解决。电话团队现在可以用 Leads 导出里的预筛字段兜底
- **拟做**：V2 做一个 `formatProjectAnswers(json, opts)` 工具函数统一格式

### V2-6 · FAQ 列表加「预览」按钮
- **出处**：PM 建议点 T3·3
- **内容**：列表每行加「预览」按钮，new tab 打开 `/faq#<slug>` 跳到患者端对应锚点
- **为什么延期**：需要 `/faq` 支持锚点定位，小改造。V2 体验增强时一起做

### V2-7 · FAQ order 字段改拖拽 UI
- **出处**：PM 补充点 T3·3
- **为什么延期**：需要引入拖拽库或手写 HTML5 drag API，V1 用数字输入 + tooltip「数字越小越靠前，默认 100」可接受

### V2-8 · 分区列表「帖子数」hover tooltip
- **出处**：PM 建议点 T4·4
- **内容**：hover 显示「已审核 X / 待审核 Y / 隐藏 Z」分项
- **为什么延期**：需要 3 次 `count` 查询聚合到单列，改造查询结构。V2 做

### V2-9 · 敏感词空态文案分两种场景
- **出处**：PM 补充点 T5（补 review 时追加）
- **内容**：区分「词库完全为空（新系统）」vs「当前筛选无命中」两种空态
- **为什么延期**：纯文案优化，不阻塞运营工作，V2 统一做所有后台空态

### V2-10 · 敏感词空态文案去掉文件路径
- **出处**：PM 补 review 时追加
- **内容**：当前「从 data/sensitive-words.txt 同步」改成「从词库模板快速导入」
- **为什么延期**：同上，V2 统一改空态文案

### V2-11 · 批量导入词库回滚
- **出处**：PM 补 review 时追加（T5 运营痛点）
- **为什么延期**：需要批量导入前先快照词库状态、导入后提供「回滚到导入前」入口。V1 运营还没到需要回滚的规模

### V2-12 · 分区封面图 / emoji 编辑入口
- **出处**：PM 补 review 时追加（T4 运营痛点）
- **为什么延期**：M5 种子有 icon 字段但后台没做编辑；V1 icon 可用默认 emoji
