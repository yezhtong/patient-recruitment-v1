# 患者社区模块 PRD V1

文档日期：2026-04-18

版本：V1.1

适用产品：临床试验患者招募网站

相关文档：[临床试验患者招募竞品调研_2026-04-17.md](</e:/Projects/03_患者招募/临床试验患者招募竞品调研_2026-04-17.md>)

## 文档摘要

本 PRD 面向“临床试验患者招募网站”的患者社区模块 V1。社区模块的定位不是泛健康论坛，而是围绕疾病认知、试验认知、病友经验和入组转化建立的互助内容层。

本版按第一版落地原则做了收缩：后台角色统一为一个 `community_operator`，前台仅保留社区首页、病种社区、帖子详情、评论、匿名发帖、FAQ、试验导流和基础审核能力。

它的目标不是替代医生建议，而是帮助患者：

1. 更容易理解临床试验
2. 更愿意留下来持续关注相关项目
3. 更自然地从内容浏览进入匹配和预筛

---

## Problem Statement

当前患者招募平台面临以下问题：

1. 患者对“临床试验”普遍陌生，容易把试验理解为高风险、无保障或不可信，导致报名意愿低。
2. 即使患者有疾病需求，也往往缺少真实病友经验，无法判断“别人是否经历过类似筛选流程”“参加试验是否麻烦”“副作用和交通成本如何”。
3. 仅靠试验列表和报名表单，很难形成留存。很多用户当前不符合某个项目，但未来可能符合，如果没有社区承接，这部分用户会直接流失。
4. 招募团队反复回答相同问题，效率低，且高频问题没有沉淀为可复用内容资产。
5. 现有平台若只做“试验黄页”，很难建立信任壁垒，也不利于形成 SEO 和自然流量。

因此，平台需要一个既服务患者理解和决策、又服务招募转化的患者社区模块。

---

## North Star Connection

平台级 North Star 假设：`Weekly Qualified Screening Starts`，即每周进入有效预筛流程的合格患者数。

社区模块对 North Star 的贡献路径：

1. 社区内容提升患者对临床试验的理解和信任，增加注册率与留存率。
2. 社区内容让患者更容易完成“自我判断”，提升预筛启动率。
3. 病种标签和社区浏览行为帮助平台理解用户关注方向，支持后续匹配和转化运营。
4. FAQ 和经验帖降低重复咨询，缩短从首次访问到预筛启动的路径。

模块级核心假设：

- 若患者在首次访问时阅读 2 篇以上高相关社区内容，其进入预筛的概率会显著高于仅浏览试验列表的患者。

## Target Users

### 主要用户

1. 患者
   目标：寻找同病种经验、理解临床试验、判断自己是否值得报名。
2. 患者家属
   目标：帮助患者做信息搜集、理解流程、降低决策焦虑。

### 次要用户

1. 社区运营后台用户
   目标：统一完成招募运营、版主管理、内容审核三类工作，沉淀 FAQ、处理违规内容、引导患者进入预筛。
2. 研究中心 / 项目方
   目标：间接获取更有准备、更有意向的患者线索。

### 角色边界

1. 匿名访客：可浏览公开内容，不可发帖、评论。
2. 注册患者/家属：可发帖、评论。
3. 社区运营后台用户：使用同一个后台角色处理招募运营、版主、审核员工作，可维护 FAQ、处理帖子和评论、配置试验推荐。

## Scope

### V1 简化原则

1. 先做“内容建立信任并导向预筛”的最小闭环，不做泛社区。
2. 先做少量重点病种社区，不追求全病种覆盖。
3. 先做统一后台角色，不拆复杂权限体系。

### In Scope

1. 社区首页。
2. 重点病种社区页。
3. 帖子详情页和评论。
4. 匿名发帖能力。
5. 官方 FAQ / 精选内容。
6. 社区内容到试验详情页、预筛页的转化入口。
7. 统一后台的内容审核和病种社区配置。
8. 最小可用数据埋点与基础看板。

### Out of Scope

1. 实时私信、群聊、直播、语音房。
   原因：实时互动和审核成本过高，不适合 V1。
2. 点赞、收藏、关注、积分、等级体系。
   原因：不影响最小闭环，先不做互动增强。
3. 社区搜索和复杂推荐算法。
   原因：V1 先依靠病种分区、置顶内容和试验挂载完成内容分发。
4. 患者之间互加联系方式、专家问诊、售药相关能力。
   原因：医疗和合规风险高，不符合平台定位。
5. 复杂权限拆分。
   原因：V1 后台统一为一个运营角色，后续再拆。

## User Stories

1. 作为患者，我想按病种查看病友讨论，这样我能判断是否有人经历过类似的治疗和试验流程。
2. 作为患者，我想匿名发帖咨询，这样我能在不暴露隐私的前提下提问敏感问题。
3. 作为患者，我想在社区帖子里直接看到相关试验入口，这样我可以在产生兴趣时马上进入预筛。
4. 作为社区运营后台用户，我想把高质量 FAQ 和帖子置顶，这样新用户能优先看到可信内容。
5. 作为社区运营后台用户，我想审核帖子和评论，这样社区可以保持安全和合规。
6. 作为招募运营，我想知道社区是否带来了预筛启动，这样我能判断社区模块是否有效。

## Functional Requirements

### 1. 社区信息架构

1. 系统必须支持社区首页，展示首批重点病种入口、官方 FAQ、精选帖子和相关试验内容。
2. 系统必须支持病种维度的一级社区，V1 只需支持平台首批重点病种的独立社区页。
3. 每个病种社区必须包含病种简介、官方 FAQ、帖子列表、相关试验推荐模块。
4. V1 帖子类型必须简化为两类：
   - 提问
   - 经验分享
5. V1 帖子列表默认按“最新”排序即可，不要求最热、关注、复杂推荐排序。

### 2. 内容发布与互动

6. 注册用户必须能够发布帖子，帖子至少包含标题、正文、病种标签、帖子类型。
7. 系统必须支持用户选择“实名昵称发布”或“匿名发布”。
8. 用户必须能够对帖子进行评论。
9. 用户必须能够删除自己仍处于公开状态的帖子与评论；删除后前台不可见，后台保留审计记录。
10. 系统必须限制单个用户在设定时间窗口内的发帖和评论频率，以降低灌水和垃圾内容。
11. V1 不要求复杂发帖模板，但经验分享类帖子可提供简单引导文案。

### 3. 结构化标签与内容组织

12. 系统必须支持帖子绑定病种标签，病种标签由后台统一维护。
13. 系统必须支持帖子绑定帖子类型。
14. V1 不要求复杂结构化标签字段，仅保留病种和帖子类型两类核心标签。
15. 帖子详情页必须展示病种标签、帖子类型和关联试验信息。

### 4. 社区内容与招募链路联动

16. 系统必须支持在帖子详情页挂载相关试验项目卡片。
17. 试验项目卡片至少包含试验标题、病种、城市、招募状态、预筛按钮。
18. 试验详情页必须展示社区入口，至少包括“病友讨论”或“经验分享”入口。
19. 对高相关帖子，系统必须展示显著的“查看相关试验”或“立即预筛” CTA。
20. 对预筛未通过或暂不符合条件的用户，系统应该推荐进入相应病种社区继续留存。

### 5. 审核与后台

21. 所有用户发布的帖子和评论必须经过内容安全校验。
22. 系统必须支持基础敏感词拦截，至少覆盖以下风险类型：
   - 联系方式外露
   - 售药/代购
   - 包入组/代办入组
   - 偏方/夸大疗效
   - 引流广告
23. 对命中高风险规则的内容，系统必须阻止发布并提示用户修改。
24. 对命中中风险规则的内容，系统必须支持进入待审状态后再发布。
25. V1 后台必须使用统一的 `community_operator` 角色，同时处理招募运营、版主、审核动作。
26. 运营后台必须支持病种社区管理，包括社区开关、病种简介、FAQ 内容、推荐试验设置。
27. 运营后台必须支持帖子和评论管理，操作至少包括通过、驳回、隐藏、精选。
28. 运营后台必须支持查看基础转化数据，至少包括社区浏览、帖子发布、社区到预筛启动。
29. 系统必须对用户与内容保留审核日志，便于复核和风控追溯。
30. 前台必须显著展示免责声明，明确社区内容不能替代医生建议。

## Non-Functional Requirements

1. 性能：社区首页、病种社区页、帖子详情页在正常网络下 P95 首屏响应时间必须小于 3 秒。
2. 安全：必须遵循平台隐私政策，不得在前台暴露手机号、微信号、身份证号等敏感信息。
3. 可用性：核心浏览、发帖、评论、审核功能月度可用性目标不低于 99.5%。
4. 审计：所有审核动作必须可追踪到操作者、时间和原因。
5. 权限：V1 只要求普通用户与统一后台运营角色的权限边界明确，后续版本再细拆权限。
6. 合规：产品文案和社区提示必须明确“社区内容仅供交流，不构成医疗建议”。
7. 无障碍：核心阅读与交互页面必须满足基础可访问性要求，包括可读字体、按钮可识别和表单错误提示。

## API Changes

### 新增前台接口

1. `GET /community/home`
   返回社区首页内容。
2. `GET /community/groups/{id}`
   返回单个病种社区信息、帖子流和相关试验。
3. `GET /community/posts/{id}`
   返回帖子详情、评论和关联试验。
4. `POST /community/posts`
   创建帖子。
5. `POST /community/posts/{id}/comments`

6. `GET /admin/community/overview`

9. `POST /admin/community/groups/{id}/update`

### 修改现有接口

10. `GET /trials/{id}`
   需增加社区内容推荐区块的数据返回。

## QA & Testing Impact

### High-risk surfaces

1. 用户注册与登录态，因为社区发帖评论需要鉴权。
2. 试验详情页，因为新增社区推荐区块，可能影响现有页面性能和布局。
3. 内容发布链路，因为敏感词拦截、待审、匿名发布会引入多种状态。
4. 后台审核链路，因为误判和漏判会直接影响内容质量与风险控制。

### New test coverage required

1. 匿名发帖与实名昵称发帖的展示差异。
2. 统一后台角色 `community_operator` 的权限测试。
3. 风险词命中后的不同处理分支。
4. 社区内容挂载试验卡片后的跳转和埋点。
5. 帖子删除、隐藏、驳回后的前后台一致性。

### Regression scope

1. 用户账户体系。
2. 试验详情页渲染与推荐模块。
3. 运营后台权限与内容管理能力。

### Edge cases to verify

1. 用户匿名发帖后再修改昵称或注销账号，历史帖子如何展示。
2. 被精选的帖子随后命中违规规则时如何下线。
3. 同一帖子既关联病种社区又关联试验项目时，关联信息是否正确展示。
4. 用户在未登录状态下点击预筛 CTA，回跳链路是否正确。
5. 运营后台隐藏帖子后，社区列表和试验详情页入口是否同步下线。

## Success Metrics

由于该模块尚未上线，基线默认为 0。以下目标以“上线后前 90 天”为观察窗口。

### 北极星支撑指标

1. 社区访客到预筛启动转化率达到 5% 以上。
2. 社区来源的预筛用户占全站预筛启动用户比例达到 10% 以上。

### 模块健康指标

1. 注册用户的社区访问率达到 30% 以上。
2. 社区访问用户的人均阅读帖子数达到 2 篇以上。
3. 首帖发布率达到 3% 以上。
4. 评论参与率达到 8% 以上。
5. 7 日社区回访率达到 15% 以上。

### 风控指标

1. 违规内容漏判率低于 2%。
2. 审核平均处理时长低于 8 小时。

## 附：V1 推荐交互流程

### 流程 1：内容消费到预筛

1. 患者通过社区首页或试验详情页进入病种社区。
2. 患者浏览帖子详情页，看到相关试验卡片和“立即预筛”按钮。
3. 患者点击 CTA。
   - 若已登录：直接进入预筛问卷。
   - 若未登录：先登录/注册，再回跳预筛问卷。
4. 系统记录社区来源和帖子来源，用于归因。

### 流程 2：匿名提问到运营承接

1. 患者选择匿名发布提问帖。
2. 系统进行风险词校验。
   - 若高风险：阻止发布并提示修改。
   - 若中风险：进入待审。
   - 若低风险：直接发布。
3. 社区运营后台用户或官方账号可在评论区给出标准化答疑，并引导到 FAQ 或预筛页。

### 流程 3：预筛失败后的社区留存

1. 患者完成预筛但未通过当前项目。
2. 系统根据病种标签推荐对应社区和 FAQ。
3. 患者返回对应病种社区继续浏览和获取后续项目信息。
