# 临床试验患者招募竞品调研

调研日期：2026-04-17

适用场景：为“临床试验患者招募网站”做零到一竞品扫描、产品构思和能力边界判断。

## 一句话结论

国内市场的主流形态更像“招募信息发布 + 微信/电话承接 + 人工协助转化”，海外成熟平台更像“试验发现 + 智能预筛 + 招募漏斗管理 + 站点协同”的一体化产品。

如果你现在要做一个零到一网站，最值得优先做的不是完整 CTMS 或 DCT 平台，而是：

1. 面向患者的试验发现与匹配
2. 面向招募运营的预筛与线索管理
3. 面向研究中心的转介和状态回传

### 国内市场观察

1. 患者侧入口高度依赖微信生态。官网往往只是信任背书页，真实转化常在公众号、小程序、电话或企业微信里完成。
2. 很多平台并不强调“算法”，真正可见的差异在病种内容页、报名表单、客服承接和研究中心协同。
3. 院内智能招募系统和患者招募网站是两条链路。前者擅长从 EMR/HIS 找人，后者擅长从公网获客；两者连接处是潜在机会。
4. 罕见病、肿瘤、慢病等高信息不对称病种，更容易形成垂直化招募产品。

### 海外市场观察

1. 海外成熟平台几乎都把“预筛问卷”做成核心能力，而不是停留在留下联系方式。
2. 很多平台同时服务患者侧和站点侧，形成双边网络：患者找试验，站点管理候选者，赞助方看漏斗数据。
3. “Plain language” 是明显差异点。把复杂方案翻译成患者听得懂的话，是转化率武器，不只是文案优化。
4. 患者导航员、呼叫中心、短信/邮件自动化依然重要，说明纯自助很难吃下全流程。

## 把竞品拆成能力层

如果把这些产品拆开，核心不是“网站/App/小程序”本身，而是下面这些能力模块：

| 能力模块 | 典型功能 | 国内常见度 | 海外常见度 | 你是否该在 V1 做 |
|---|---|---|---|---|
| 试验目录与搜索 | 按疾病、城市、机构、阶段筛选 | 高 | 高 | 应做 |
| 患者教育内容 | 试验科普、常见疑问、风险解释 | 中 | 高 | 应做 |
| 患者社区 | 病种社区、经验交流、问答、试验经历分享、互助答疑 | 中 | 中到高 | 应做 |
| 在线预筛 | 多步问卷、必填条件校验、适应症判断 | 中 | 很高 | 强烈建议做 |
| 患者档案库 | 反复复用患者资料、二次匹配 | 低到中 | 高 | 可做轻量版 |
| 人工导航/客服 | 电话回访、企业微信、短信提醒 | 很高 | 高 | 应做 |
| 研究中心工作台 | 查看线索、更新状态、预约筛查 | 低到中 | 高 | 应做基础版 |
| 数据回传与漏斗分析 | 点击、报名、预筛通过、到院、入组 | 低 | 高 | 强烈建议做 |
| 院内数据匹配 | HIS/EMR/NLP 自动筛选 | 少数平台具备 | 中到高 | 暂不建议 V1 自建 |
| 试验执行延伸 | eConsent、ePRO、远程访视 | 少数平台具备 | 中到高 | 不建议 V1 就做 |

## 对你做网站的产品启发

### 建议的 V1 定位

先把自己定义成“患者招募与预筛平台”，不要一开始做成“大而全的临床试验数字化平台”。

更具体一点，V1 最适合解决这四个问题：

1. 患者找不到合适试验
2. 患者缺少可信的病友经验和临床试验认知支持
3. 研究中心拿到的线索不够准
4. 招募团队无法看清每个项目的转化漏斗

### 建议的 V1 功能骨架

#### 1. 患者侧

- 试验搜索页：按疾病、城市、年龄、招募状态筛选
- 试验详情页：把医学术语翻成患者语言，突出“我是否可能符合”
- 在线预筛问卷：分基础资料、疾病条件、禁忌条件三层
- 报名与咨询：表单 + 微信/电话 + 短信确认
- 我的匹配：同一患者可被多个项目重复匹配
- 患者社区：按病种/治疗阶段进入讨论区，查看他人试验经历、提问、收藏内容

#### 2. 招募运营侧

- 线索池：新线索、待联系、预筛通过、预约中、失败原因
- 分配机制：按病种/城市/研究中心分配给不同运营或 CRC
- 回访记录：电话、微信、短信、备注
- 失败原因结构化：不符合、失联、拒绝、已入其他试验

#### 3. 研究中心侧

- 项目页面维护：招募标准、中心信息、联系人、更新时间
- 候选者接收：查看候选者资料和预筛结果
- 状态回传：是否预约、是否到院、是否入排
- 数据看板：项目级转化率和来源渠道

### 患者社区模块设计（V1 必做）

我建议把患者社区定义为“围绕疾病与试验旅程的互助社区”，而不是泛化健康论坛。它在 V1 就值得做，因为它同时解决四件事：

1. 建立信任，让患者在正式报名之前先理解临床试验是什么、别人是怎么参与的。
2. 提升留存，让暂时不符合当前项目的人仍然愿意留在平台。
3. 积累内容资产，让平台逐步拥有 SEO、问答、病种经验内容。
4. 反哺招募，让社区讨论中的高意向患者自然进入匹配和预筛。

#### 典型功能

- 病种社区首页：按癌种、罕见病、慢病等建立一级社区。
- 主题分区：建议至少有“治疗经历”“临床试验问答”“检查与副作用”“入组经验”四类。
- 发帖与问答：支持文字发帖、匿名发帖、评论、点赞、收藏。
- 试验经历卡片：鼓励用户按模板分享“我为什么报名”“筛选流程”“交通/时间成本”“常见顾虑”。
- 内容搜索：支持按病种、关键词、试验编号、药物名搜索。
- 社区内容到招募页跳转：帖子内可挂相关试验、病种招募页、预筛入口。
- 官方内容置顶：由平台运营发布“临床试验科普”“报名避坑”“流程说明”等权威帖。
- 私信或群聊先不做：V1 先避免强实时社交和高审核成本功能。

#### V1 推荐的信息架构

- 社区首页：展示热门病种社区、最新讨论、精选试验经验。
- 病种社区页：展示病种介绍、置顶 FAQ、帖子流、相关招募项目。
- 帖子详情页：正文、评论区、相关推荐、预筛 CTA。
- 我的社区：我的发帖、评论、收藏、关注病种。

#### V1 关键交互设计

- 用户浏览任一试验详情页时，可看到“相似患者怎么讨论这个病种/治疗”的社区入口。
- 用户阅读社区内容 2 到 3 篇后，引导其完成“我的基本情况”，用于后续项目匹配。
- 用户发与试验相关的帖子时，可引导补充“疾病、分期、城市、是否做过基因检测”等结构化标签。
- 用户看到高相关帖子时，可一键进入对应试验的预筛问卷。

#### V1 内容机制

- 内容生产以 UGC + 官方 PGC 混合为主。
- 冷启动阶段，优先由运营补 50 到 100 篇高质量基础内容，包括常见问题、报名经历、检查流程、交通住宿说明。
- 每个病种社区先做 1 名到 2 名官方版主角色，负责答疑、审核、整理精华帖。
- 所有社区内容都要显著声明“不能替代医生建议”，避免被理解为医疗建议。

#### V1 审核与风控要求

- 支持匿名昵称，不默认公开手机号、微信号、身份证号等敏感信息。
- 发帖时拦截联系方式外露、广告引流、偏方、虚假疗效宣传。
- 对“求药”“售药”“代办入组”“包入组”等高风险内容做自动拦截。
- 对涉及严重不良事件、自杀倾向、极端情绪的内容设人工升级处理流程。

#### V1 与招募主链路的关系

- 社区不是独立产品，而是招募漏斗顶部和中部的信任层。
- 社区入口应出现在首页、病种页、试验详情页、报名成功页。
- 招募运营可以把高频咨询沉淀成社区 FAQ，减少重复人工答疑。
- 预筛未通过的用户，可推荐进入相应病种社区继续留存。

#### 成功指标

- 社区访客到注册转化率
- 社区用户到预筛启动率
- 社区用户到报名提交率
- 首帖发布率
- 评论参与率
- 次日/7日回访率
- 内容举报率与审核通过率

#### 可参考的海外社区产品思路

- `PatientsLikeMe`：强调患者真实经历沉淀和结构化健康旅程。
- `Inspire`：强调安全、匿名、病种分群和患者组织合作。
- `Smart Patients`：强调“临床试验社区”与疾病讨论的自然连接。

### 建议你暂时不要优先做的东西

1. 自建院内病历 NLP 匹配
2. 复杂的 eConsent/ePRO/远程访视
3. 全量 CRO/CTMS 项目管理功能
4. 过早做成开放 marketplace，让所有研究中心自助发布并完全自运营

这些功能不是不重要，而是它们更适合在你验证了患者流量和项目转化之后再加。

## 你可以重点学习的 5 个竞品套路

1. 学 `Power`：把试验详情写成患者看得懂的消费级页面，而不是医学登记页。
2. 学 `Antidote`：做标准化预筛逻辑，并支持白标嵌入合作方页面。
3. 学 `Trialbee / AutoCruitment`：把投放来源到入组的漏斗串起来，运营可见。
4. 学 `募邦 / 药试圈`：利用中国用户习惯的公众号、小程序、电话协同承接。
5. 学 `WeTrial-RMS`：给研究中心一个最小可用工作台，否则线索交付后会失控。

## 一个可行的差异化切口

如果你不想只做“试验信息黄页”，可以考虑下面三个切口之一：

### 方案 A：重患者体验

定位：最容易看懂、最容易报名、最有人协助的患者招募网站。

适合：肿瘤、慢病、罕见病等信息门槛高的病种。

### 方案 B：重运营效率

定位：帮助招募团队把流量、预筛、回访、到院、入组全部打通。

适合：你已有药企/CRO/研究中心项目资源，希望提升转化效率。

### 方案 C：重垂直病种社区

定位：先做一个病种深、信任强、内容强的垂直招募入口，再扩病种。

适合：罕见病、肿瘤亚型、生殖、神经退行性疾病等高匹配难度赛道。

## 我对你的初步建议

如果你现在还在产品构思阶段，我会建议：

1. 先不要思考“要不要做 App”，先把 Web + 微信承接跑通。
2. 先不要思考“能不能自动匹配所有试验”，先把 20 到 50 个高质量项目的人工+规则匹配做准。
3. 先把“预筛问卷 + 线索池 + 中心回传”这条最小闭环跑起来。
4. 先挑一个病种或一个合作场景做样板，而不是全病种铺开。
