## 文档摘要

本文件用于定义“临床试验患者招募网站”V1 的整体范围，覆盖三部分：

1. V1 信息架构
2. V1 功能清单
3. 可继续展开的总 PRD 骨架

本版是在以下约束下收敛后的可落地骨架：

1. 患者社区是 V1 强需求，但仍然是辅助主链路的模块，不替代“找试验 + 预筛”主体。
2. 社区中必须出现预筛入口，形成“内容信任 -> 预筛转化”路径。
3. 患者账户进入 V1，但只做轻量账户，不做复杂会员体系。
4. 自定义预筛表单进入 V1，但必须限制在可快速实现的范围内。
5. 整体范围需要适合短周期 `vibe coding`。

补充说明：

1. 本文中的“病种”在实现层统一按“病种 / 适应症”理解。
2. 标签体系、试验筛选、社区分区都必须兼容医美相关适应症，而不只覆盖传统疾病分类。

## V1 设计前提

### 业务前提

1. V1 首先是患者招募和预筛转化工具，不是完整临床试验数字化平台。
2. V1 的核心主链路仍然是“浏览试验 -> 理解试验 -> 提交预筛 -> 运营处理线索”。
3. 患者社区在 V1 中承担信任建立、留存和内容导流角色，是强需求，但不是主工作台。
4. 患者账户在 V1 中主要服务于资料复用、报名记录和社区参与。
5. 研究中心协同在 V1 中先由平台运营人工承接和记录，不单独建设复杂研究中心门户。

### vibe coding 前提

1. 尽量做成 `一个公开网站 + 一个后台管理台`。
2. 使用单数据库、单应用主线、少量固定状态流转。
3. 患者账户只保留最小必要能力：登录、基本资料、我的报名、我的社区内容。
4. 社区只保留最小互动闭环：分区、发帖、评论、关联试验、预筛 CTA，不做复杂社交。
5. 自定义预筛表单只支持有限字段类型和有限逻辑，不做通用低代码表单引擎。

## V1 目标

### 业务目标

1. 建立一个可公开获客的临床试验招募网站。
2. 让患者从浏览试验走到提交预筛形成闭环。
3. 让患者在提交前后都能通过 FAQ 和社区内容建立信任并继续留存。
4. 让运营可以统一查看、跟进、分配和更新线索状态。
5. 让平台能够看清基础漏斗：访问、详情浏览、预筛启动、预筛提交、运营跟进。

### 用户目标

1. 患者能快速找到和自己疾病相关的试验。
2. 患者能看懂试验，而不是只看到登记平台式术语。
3. 患者能注册账户，复用资料、查看报名记录、参与社区。
4. 患者能在社区中看到相关试验和预筛入口。
5. 运营能快速判断线索价值并进行后续跟进。

### V1 不追求的目标

1. 不做完整 CTMS / CRO 项目管理平台。
2. 不做院内 EMR/HIS/NLP 智能筛查。
3. 不做研究中心独立工作台。
4. 不做复杂社交体系，如私信、群聊、直播、等级体系。
5. 不做通用低代码流程引擎。
6. 不做 eConsent、ePRO、远程访视。

---

## North Star 与关键指标

### 平台级 North Star

`Weekly Qualified Prescreen Submissions`

定义：每周完成提交、信息完整、可被运营处理的有效预筛单数。

### 核心支撑指标

1. 首页到试验详情点击率
2. 试验详情到预筛启动率
3. 预筛启动到提交率
4. 预筛提交到运营首次跟进率
5. 社区内容到预筛启动率
6. 注册转化率
7. 登录患者资料复用率

## 目标用户与角色

| 角色 | 类型 | 核心任务 | V1 权限/能力 |
|---|---|---|---|
| 匿名访客 | 患者侧 | 浏览试验、阅读 FAQ、浏览社区内容 | 可浏览公开页面，不可发帖评论，不建议直接完成最终提交 |
| 注册患者/家属 | 患者侧 | 提交预筛、查看报名记录、参与社区、维护资料 | 可登录、提交预筛、查看我的报名、发帖评论 |
| 运营用户 `operator` | 平台后台 | 维护试验、配置预筛表单、查看线索、维护内容、审核社区 | 使用统一后台角色处理试验、表单、线索、FAQ、社区内容 |
| 管理员 `admin` | 平台后台 | 管理运营账号、标签、系统配置 | V1 仅少量内部人员使用 |

### 角色收敛原则

1. 患者侧只有“匿名访客”和“注册患者/家属”两类角色。
2. 后台只保留 `operator` 与 `admin` 两类角色。
3. 社区审核、FAQ 管理、线索处理都由 `operator` 承接。
4. 研究中心协同信息由 `operator` 在后台记录，不单独提供研究中心登录入口。

## V1 产品原则

1. 主链路优先：先做“找试验 -> 预筛 -> 运营处理”。
2. 社区强需求但不喧宾夺主：服务信任、留存和导流。
3. 账户轻量化：以手机号登录和基础资料复用为核心。
4. 表单定制受控：允许项目级差异，但控制组件种类和逻辑复杂度。
5. 人工优先：先允许运营人工判断与协同，不做复杂自动化。

## 二、V1 关键用户旅程

## 2.1 患者浏览试验并提交预筛

1. 患者进入首页或找试验页。
2. 患者按疾病、城市浏览试验列表。
3. 患者打开试验详情页。
4. 患者点击“立即预筛”。
5. 若未登录，先进入登录/注册；若已登录，直接进入表单。
6. 患者填写该项目预筛表单并提交。
7. 系统进入成功页，提示后续会有运营联系。

## 2.2 患者通过社区建立信任后进入预筛

1. 患者浏览社区首页或病种社区页。
2. 患者进入帖子详情页。
3. 帖子中出现相关试验卡片和“立即预筛” CTA。
4. 患者点击进入相关试验详情页或直接进入预筛表单。
5. 患者完成提交。

## 2.3 已登录患者复用资料进行再次报名

1. 患者登录账户。
2. 患者进入新的试验详情页。
3. 系统自动带出基础资料到预筛表单。
4. 患者补充项目特有字段并提交。

## 2.4 运营处理线索

1. 运营登录后台工作台。
2. 运营在“新增线索”中查看新提交的预筛单。
3. 运营打开线索详情，查看患者资料、项目表单答案和来源。
4. 运营填写跟进记录、更新状态。
5. 运营根据情况记录推荐中心或中心反馈备注。

## 3.5 自定义预筛表单的 V1 收敛方案

为了兼顾“项目可定制”和“vibe coding 可实现”，V1 的项目级自定义预筛只允许以下范围：

### 支持的字段类型

1. 单行文本
2. 多行文本
3. 单选
4. 多选
5. 是/否
6. 数字
7. 日期

### 支持的字段属性

1. 字段标题
2. 字段说明
3. 是否必填
4. 选项列表
5. 排序顺序
6. 是否属于基础资料字段

### V1 不支持的复杂能力

1. 多层级条件跳转
2. 可视化拖拽式表单搭建
3. 复杂表达式校验
4. 动态联动字典
5. 文件上传字段

### 推荐实现方式

1. 以固定字段组件库为基础做项目级配置。
2. 以 JSON 配置或结构化表设计存储表单。
3. 患者基础资料字段和项目特有字段分层渲染。

## 3.6 明确不进入 V1 的功能

| 模块 | 功能 | 原因 |
|---|---|---|
| 社区 | 私信、群聊、直播、积分、等级、关注关系 | 明显增加社交复杂度和审核成本 |
| 研究中心 | 独立登录入口和候选者处理台 | 会形成第二套后台系统，不利于短期交付 |
| 账户 | 实名认证、复杂隐私授权流、长期会员体系 | 会显著增加账户复杂度 |
| 预筛 | 通用低代码表单引擎 | 会增加大量后台配置和前端渲染复杂度 |
| 自动化 | 自动分配中心、自动短信编排、自动风控工作流 | V1 可由运营人工完成 |

## 六、最适合 vibe coding 的实现边界

这部分不是产品功能，而是帮助后续根据 PRD 做实现时减少失控。

## 6.1 建议的实现边界

1. 一个公开站点，一个后台站点。
2. 一个主数据库。
3. 一个患者账户体系，一个运营后台体系。
4. 试验详情使用统一页面模板。
5. 社区使用统一帖子模板和统一评论模型。
6. 预筛表单使用受控字段组件库。

## 6.2 不建议在首版加入的技术复杂点

1. 多角色、多权限矩阵。
2. 通用低代码表单引擎。
3. 实时通知系统。
4. 社区推荐算法。
5. 文件上传审核流。
6. 独立研究中心系统。

## 6.3 建议的首版产品边界

1. 首批只支持少量重点病种 / 适应症。
2. 社区首期只做病种 / 适应症分区，不做复杂社交网络。
3. 预筛字段组件先限制在 7 类以内。
4. 研究中心协同先通过线索备注和状态字段实现。
5. 患者账户先使用手机号验证码登录。
