## 文档摘要

本 PRD 用于定义“临床试验患者招募网站”V1 的完整产品方案，覆盖以下内容：

1. 项目背景与目标。
2. 用户角色与产品边界。
3. 公开网站与运营后台的信息架构。
4. 核心用户旅程与功能需求。
5. 页面清单与路由建议。
6. 核心状态机、数据模型与接口边界。
7. 埋点、指标、测试重点与上线边界。

本版在前序讨论基础上，明确以下关键结论：

1. V1 的主链路始终是 `找试验 -> 看懂试验 -> 提交预筛 -> 运营跟进线索`。
2. 患者社区是 V1 强需求模块，但定位为信任层、留存层和导流层，不替代主链路。
3. 社区页面必须出现预筛入口，并与试验详情页、项目级预筛表单形成闭环。
4. 患者账户是 V1 范围内模块，但保持轻量化，只服务于登录、资料复用、报名记录和社区参与。
5. 自定义预筛表单是 V1 范围内模块，但只做“项目级受控配置”，不做通用低代码表单平台。
6. 后台采用统一运营台，只保留 `operator` 与 `admin` 两类角色，不建设研究中心独立后台。
7. 病种体系在实现上统一按“病种 / 适应症”理解，必须兼容传统疾病与医美相关适应症。
8. 整体范围必须适合短周期 `vibe coding` 落地，优先做单网站、单后台、单数据库、固定状态流转。

## Problem Statement

结合竞品调研与当前目标，临床试验患者招募网站要解决的问题主要有以下几类：

1. 患者很难找到适合自己的试验项目。
   现有公开登记平台偏监管和学术表述，普通患者难以直接理解。
2. 患者即使找到了项目，也看不懂试验内容。
   很多平台停留在“登记信息展示”，缺少患者友好解释、流程说明和风险理解支持。
3. 患者缺少可信的试验认知和病友经验支持。
   仅靠试验列表和报名按钮，很难建立信任，也难以承接暂未报名的人群。
4. 每个项目的预筛问题存在差异，但又不能让实现复杂度失控。
   完全固定问卷不够用，完全通用表单引擎又过重，不适合 V1。
5. 患者重复报名时资料重复填写，转化体验差。
   如果没有患者账户和资料复用能力，提交率会下降。
6. 运营缺少统一后台来承接试验上线、表单配置、线索跟进和内容审核。
   没有统一运营台，项目迭代效率低，线索处理容易断裂。
7. 如果一开始把系统做成复杂多端、多角色、多工作流平台，会超出短周期可实现范围。
   V1 需要在“可用”和“可实现”之间找到平衡。

因此，V1 的目标不是建设完整临床数字化平台，而是建设一个能够真正跑通患者招募闭环的轻量可用系统。

## Market Insight Summary

基于竞品调研，可以得出三点对 V1 直接有用的判断：

1. 国内主流工具更偏“信息发布 + 微信 / 电话承接 + 人工转化”。
   说明最小闭环可以先不做过重的自动化系统，但必须把信息展示和线索承接做好。
2. 海外成熟工具更偏“试验发现 + 在线预筛 + 站点协同 + 漏斗分析”。
   说明中长期正确方向是闭环化、结构化、数据化，而不是只做信息黄页。
3. 对患者来说，信任层非常关键。
   FAQ、患者教育内容、病种社区和患者友好文案并不是附属内容，而是招募转化前置条件。

V1 采取的产品策略是：

1. 先跑通国内最现实的闭环。
2. 在公开网站侧补足患者友好解释和患者社区。
3. 在后台侧补足结构化预筛、线索池与基础数据。
4. 不在 V1 引入 CTMS、院内系统、复杂工作流和独立研究中心后台。

## Product Goals

### 业务目标

1. 搭建一个可公开获客的临床试验患者招募网站。
2. 让患者从浏览试验到提交预筛形成闭环。
3. 让患者在报名前后都能通过 FAQ 与社区内容建立信任并持续留存。
4. 让运营可以统一管理试验、预筛表单、线索、FAQ、社区内容和基础数据。
5. 让平台能够追踪最基础的转化漏斗与内容导流效果。

### 用户目标

1. 患者能够快速找到相关试验。
2. 患者能够看懂试验，不再只面对登记平台式术语。
3. 患者能够登录账户并复用资料。
4. 患者能够在社区里获得病种相关经验，并从社区直接进入预筛。
5. 运营能够快速上线项目、处理线索、审核内容。

### V1 不追求的目标

1. 不做完整 CTMS / CRO 项目管理平台。
2. 不做院内 EMR / HIS / NLP 智能筛查。
3. 不做研究中心独立工作台。
4. 不做复杂社交体系，如私信、群聊、直播、积分、等级。
5. 不做通用低代码流程引擎和通用低代码表单平台。
6. 不做 eConsent、ePRO、远程访视。
7. 不做 App 和完整小程序版本。

## North Star Connection

平台级 North Star：

`Weekly Qualified Prescreen Submissions`

定义：每周完成提交、信息完整、可被运营处理的有效预筛单数。

### 支撑指标

1. 首页到试验详情点击率。
2. 试验详情到预筛启动率。
3. 预筛启动到提交率。
4. 预筛提交到运营首次跟进率。
5. 社区内容到预筛启动率。
6. 注册转化率。
7. 已登录患者资料复用率。

### 核心假设

1. 如果试验详情页足够患者友好，详情页到预筛启动率会明显高于单纯登记信息展示。
2. 如果项目级预筛表单能体现项目差异，同时复用患者基础资料，启动到提交率会提升。
3. 如果社区内容与 FAQ 能建立信任并导向预筛，患者会更愿意进入预筛而不是直接流失。
4. 如果后台能统一承接线索和内容，试验上线效率与线索处理效率会提升。

## Design Principles

### 业务原则

1. 主链路优先。
   所有设计都服务于“找试验 -> 预筛 -> 跟进”。
2. 社区强需求但不喧宾夺主。
   社区是辅助转化层，不是泛社交产品。
3. 患者语境优先。
   文案和页面信息优先回答患者最关心的问题，而不是复制监管平台术语。
4. 账户轻量化。
   账户只服务于资料复用、历史查看和社区参与。
5. 表单可配置但必须受控。
   允许项目差异，限制复杂度。
6. 人工优先于自动化。
   先允许运营人工判断和协同，不做复杂自动流程。

### vibe coding 原则

1. 单一公开网站 + 单一运营后台。
2. 单数据库、单应用主线。
3. 固定页面结构、固定状态流转。
4. 固定字段组件库，而不是自由搭建器。
5. 单一内容审核工作台，而不是复杂运营中台。
6. 先做可运行的闭环，再做增强能力。

## Target Users

| 角色 | 类型 | 核心任务 | V1 权限 / 能力 |
|---|---|---|---|
| 匿名访客 | 患者侧 | 浏览试验、FAQ、社区公开内容 | 可浏览公开页面，不可发帖评论，建议在提交预筛前登录 |
| 注册患者 / 家属 | 患者侧 | 提交预筛、维护资料、查看报名记录、参与社区 | 可登录、提交预筛、查看我的报名、发帖评论 |
| 运营用户 `operator` | 后台 | 维护试验、配置预筛表单、处理线索、维护 FAQ、审核社区 | 使用统一后台处理试验、表单、线索、FAQ、社区内容 |
| 管理员 `admin` | 后台 | 管理后台账号、标签、敏感词和系统配置 | 拥有 `operator` 全部能力，并管理系统配置 |

### 角色收敛原则

1. 患者侧只有“匿名访客”和“注册患者 / 家属”两类角色。
2. 后台只保留 `operator` 与 `admin` 两类角色。
3. 社区审核、FAQ 管理、线索处理都由 `operator` 承接。
4. 研究中心协同信息由 `operator` 在后台记录，不单独提供研究中心登录入口。

## Key User Journeys

### Journey 1：患者通过试验列表进入预筛

1. 匿名访客进入首页或试验列表页。
2. 按病种 / 适应症和城市筛选试验。
3. 打开试验详情页，阅读患者友好说明。
4. 点击 `立即预筛`。
5. 若未登录，进入登录 / 注册页；若已登录，直接进入预筛表单。
6. 填写项目级预筛表单并提交。
7. 进入提交成功页，并看到 FAQ / 社区推荐。
8. 后台生成线索，进入线索池。

### Journey 2：患者通过社区建立信任后进入预筛

1. 患者浏览社区首页或病种 / 适应症社区页。
2. 阅读帖子详情。
3. 在帖子页看到相关试验卡片和 `立即预筛` CTA。
4. 点击进入试验详情页或直接进入当前项目预筛表单。
5. 登录后完成提交。
6. 系统记录社区来源和帖子来源。

### Journey 3：已登录患者复用资料再次报名

1. 已登录患者进入新的试验详情页。
2. 点击 `立即预筛`。
3. 系统自动带出基础资料字段。
4. 患者仅补充项目特有字段。
5. 提交成功后，在“我的报名”中可见该记录。

### Journey 4：运营上线试验并配置预筛表单

1. `operator` 登录后台。
2. 在试验管理中创建试验或复制已有试验。
3. 编辑患者友好说明、城市、病种 / 适应症、招募状态等。
4. 进入预筛表单配置页，配置基础资料字段与项目字段。
5. 预览并发布预筛表单。
6. 发布试验后，公开网站可见该试验与预筛入口。

### Journey 5：运营处理线索

1. `operator` 登录后台工作台。
2. 在“今日新增线索”进入线索池。
3. 打开线索详情，查看患者基础资料、项目答案与来源。
4. 记录电话 / 微信 / 其他沟通情况。
5. 更新线索状态。
6. 若需要，手动记录推荐中心与中心反馈备注。

### Journey 6：运营审核社区内容

1. 已登录患者发帖或评论。
2. 系统完成内容安全校验。
3. 内容进入待审核或直接公开。
4. `operator` 在社区审核页进行通过、驳回、隐藏、精选操作。
5. 对精选内容配置相关试验与预筛导流。

## Functional Requirements

## A. 公开网站

### 1. 全局站点能力

1. 系统必须提供统一公开网站导航。
2. 公开网站必须同时适配桌面端与移动端浏览器。
3. 试验详情页、FAQ 详情页、帖子详情页必须具备可分享链接。
4. 全站必须显著提供 `开始找试验` 与 `立即预筛` 入口。
5. 全站必须提供基础免责声明，明确平台内容不能替代医生建议。
6. 所有公开页面必须支持病种 / 适应症标签口径，而不是仅传统疾病口径。

### 2. 首页

7. 首页必须完成平台价值说明与流量承接。
8. 首页 Hero 区必须包含：
   - 主标题
   - 副标题
   - 主 CTA `开始找试验`
   - 次 CTA `先了解临床试验`
9. 首页必须展示首批重点病种 / 适应症入口。
10. 首页必须展示重点试验推荐。
11. 首页必须展示平台流程说明，至少解释：
   - 找到相关项目
   - 完成资格预筛
   - 等待运营联系
12. 首页必须展示 FAQ 入口。
13. 首页必须展示社区入口。
14. 首页重点试验和重点病种入口允许由后台手动配置。

### 3. 试验列表页

15. 试验列表页必须支持按病种 / 适应症筛选。
16. 试验列表页必须支持按城市筛选。
17. 试验列表页必须支持按招募状态筛选。
18. 试验列表页必须展示结果数量。
19. 试验卡片至少必须展示：
   - 试验标题
   - 病种 / 适应症标签
   - 城市 / 研究地点
   - 招募状态
   - 一句话摘要
   - CTA `查看详情`
20. 试验列表页若无结果，必须提供空状态说明和继续浏览 FAQ / 社区的入口。

### 4. 试验详情页

21. 试验详情页必须使用患者友好语言解释项目。
22. 试验详情页至少必须包含以下模块：
   - 试验标题区
   - 简短摘要
   - 适合人群
   - 不适合人群
   - 参与流程说明
   - 城市 / 研究地点
   - FAQ 相关内容
   - 社区相关入口
   - 主 CTA `立即预筛`
23. 试验详情页必须支持展示社区入口，至少包含“病友讨论”或“经验分享”入口。
24. 试验详情页必须支持展示相关 FAQ。
25. 当试验未处于 `招募中` 或未发布时，前台不得暴露可提交的预筛入口。
26. 若试验暂停招募或已关闭，详情页必须展示对应状态说明。

### 5. 登录 / 注册页

27. 系统必须提供统一登录 / 注册页。
28. 患者端 V1 必须支持轻量账户体系。
29. 登录 / 注册页必须承接以下来源：
   - 试验详情页点击预筛
   - 社区发帖
   - 社区评论
   - 我的报名
30. 登录后系统必须支持回跳到用户原始目标页。
31. 产品目标方案以手机号登录为主。
32. 为适配短周期开发，技术实现层可先采用可控的简化认证方案，但对用户暴露的产品能力仍按“手机号登录 / 注册”设计。

### 6. 基本资料页

33. 已登录患者必须能维护基础资料。
34. 基本资料页至少应包含：
   - 姓名
   - 性别
   - 年龄
   - 手机号
   - 所在城市
   - 主要病种 / 适应症
35. 基本资料页保存后必须可复用于后续预筛。
36. 基本资料页不要求扩展为复杂健康档案。

### 7. 项目级预筛表单页

37. 每个试验必须加载自己的项目级预筛表单。
38. 预筛表单必须分为两类字段：
   - 基础资料字段
   - 项目特有字段
39. 已登录患者进入预筛表单时，系统必须自动预填基础资料字段。
40. V1 必须支持以下字段类型：
   - 单行文本
   - 多行文本
   - 单选
   - 多选
   - 是 / 否
   - 数字
   - 日期
41. V1 不支持文件上传。
42. V1 不支持多层复杂条件跳转。
43. 每个字段必须支持：
   - 字段编码
   - 标题
   - 说明
   - 类型
   - 是否必填
   - 排序
44. 选择类字段必须支持选项配置。
45. 数字字段必须支持最小值 / 最大值。
46. 日期字段必须支持日期范围。
47. 表单提交前必须完成必填校验和基础格式校验。
48. 表单页必须明确区分“基础资料”和“项目相关问题”。
49. 表单页必须展示项目名称，避免患者不知道自己在为哪个项目提交。
50. 社区和试验详情页中的 `立即预筛` 必须跳转到对应项目当前生效的预筛表单。
51. 针对传统病种项目与医美适应症项目，后台必须支持配置不同项目字段。
52. 医美相关适应症项目示例问题可包括：
   - 是否存在某类皮肤困扰
   - 困扰持续时间
   - 是否接受过相关治疗
   - 近 6 个月是否进行过某类医美项目

### 8. 预筛提交成功页

53. 预筛提交成功页必须提示后续会有运营联系。
54. 预筛提交成功页必须提供 FAQ 入口。
55. 预筛提交成功页必须提供社区入口。
56. 若后续判断暂不符合条件，也允许通过成功页导向社区做留存。

### 9. 我的报名页

57. 已登录患者必须能查看自己的报名记录。
58. 每条报名记录至少必须展示：
   - 试验名称
   - 提交时间
   - 当前状态
   - 试验详情入口
59. 患者端状态必须保持用户友好，不直接暴露过于内部化的后台状态名称。
60. “我的报名”页不要求支持撤回、编辑历史提交答案。

### 10. FAQ 列表页与详情页

61. 系统必须提供 FAQ 列表页。
62. FAQ 必须支持分类展示。
63. FAQ 卡片至少展示标题、摘要和分类。
64. FAQ 详情页必须支持展示相关试验或相关病种 / 适应症入口。
65. FAQ 在首页、试验详情页、社区页必须可被复用展示。

### 11. 社区首页

66. 系统必须提供社区首页。
67. 社区首页必须展示重点病种 / 适应症社区入口。
68. 社区首页必须展示精选帖子。
69. 社区首页必须展示官方 FAQ 入口。
70. 社区首页必须展示推荐试验或社区导流项目。

### 12. 病种 / 适应症社区页

71. 系统必须支持按病种 / 适应症维度建立一级社区页。
72. 每个病种 / 适应症社区页至少必须包含：
   - 社区名称
   - 社区简介
   - 官方 FAQ
   - 帖子列表
   - 相关试验推荐
   - `发帖` 按钮
   - `立即预筛` 入口
73. 病种 / 适应症社区页必须支持展示相关试验卡片。
74. 社区页帖子列表默认按最新排序。
75. V1 不要求复杂推荐排序、关注排序和热度排序。

### 13. 帖子详情页

76. 帖子详情页必须展示标题、正文、作者展示名、病种 / 适应症标签、帖子类型。
77. 帖子详情页必须展示评论列表。
78. 帖子详情页必须展示相关试验卡片。
79. 帖子详情页必须提供显著的 `查看相关试验` 或 `立即预筛` CTA。
80. 若帖子作者选择匿名发布，前台必须以匿名身份展示。

### 14. 发帖与评论

81. 已登录患者必须能够发帖。
82. 帖子至少必须包含：
   - 标题
   - 正文
   - 病种 / 适应症标签
   - 帖子类型
83. 帖子类型在 V1 中必须简化为两类：
   - 提问
   - 经验分享
84. 系统必须支持用户选择实名昵称发布或匿名发布。
85. 已登录患者必须能够评论帖子。
86. 用户必须能够删除自己仍处于公开状态的帖子与评论。
87. 系统必须限制单个用户在设定时间窗口内的发帖和评论频率。
88. V1 不要求点赞、收藏、关注、积分、等级体系。

### 15. 我的帖子 / 评论页

89. 已登录患者必须能查看自己在社区里的帖子和评论记录。
90. 该页面至少展示：
   - 标题或摘要
   - 类型
   - 发布时间
   - 当前状态
   - 对应详情入口
91. 当前状态至少包括：
   - 已发布
   - 审核中
   - 未通过

### 16. 联系我们页

92. 系统必须提供联系方式页。
93. 联系方式页至少展示平台联系方式与咨询说明。
94. V1 可先使用固定联系方式，不要求复杂在线客服系统。

## B. 统一运营后台

### 17. 后台登录与权限

95. 系统必须支持 `operator` 与 `admin` 通过统一登录页进入后台。
96. V1 必须只保留 `operator` 与 `admin` 两类后台角色。
97. `operator` 必须可访问：
   - 工作台
   - 试验管理
   - 预筛表单配置
   - 线索池
   - FAQ 管理
   - 社区分区配置
   - 社区审核
   - 基础统计
98. `admin` 必须拥有 `operator` 全部能力，并额外可访问：
   - 后台账号管理
   - 敏感词配置
   - 病种 / 适应症标签配置
   - 城市配置
99. V1 不要求复杂权限矩阵。
100. 所有关键后台操作都必须记录审计日志。

### 18. 工作台首页

101. 工作台首页必须展示今日新增线索数。
102. 工作台首页必须展示待联系线索数。
103. 工作台首页必须展示待审核帖子 / 评论数。
104. 工作台首页必须展示当前招募中试验数。
105. 工作台首页必须展示最近 7 日预筛启动数与提交数。
106. 工作台首页必须提供快捷入口，至少包括：
   - 新建试验
   - 查看新线索
   - 查看待审核内容
   - 新建 FAQ
107. V1 不支持自由拖拽工作台模块。

### 19. 试验管理

108. 后台必须支持试验列表页。
109. 试验列表必须支持按试验名称、病种 / 适应症、城市、招募状态筛选。
110. 试验列表行内至少展示：
   - 试验名称
   - 病种 / 适应症
   - 招募城市
   - 招募状态
   - 公开状态
   - 当前表单状态
   - 最近更新时间
111. 后台必须支持新建试验和编辑试验。
112. 试验编辑页至少必须支持以下字段：
   - 试验标题
   - 简短摘要
   - 患者友好型试验说明
   - 适合人群
   - 不适合人群
   - 参与流程说明
   - 城市 / 研究地点
   - 招募状态
   - 公开状态
   - 病种 / 适应症标签
113. 试验必须支持关联 FAQ 内容。
114. 试验必须支持关联病种 / 适应症社区。
115. 试验必须支持配置是否在首页或社区中作为推荐项目展示。
116. V1 应支持复制已有试验作为新试验的起点。

### 20. 项目级预筛表单配置

117. 每个试验必须存在“预筛表单配置”入口。
118. 后台必须允许从固定字段组件库中配置基础资料字段与项目字段。
119. 后台必须支持设置字段顺序、是否必填、说明文字和选项。
120. 后台必须支持表单预览。
121. 后台必须支持表单发布。
122. 一个试验同一时刻只能存在一个当前生效的表单版本。
123. 历史表单版本和历史答案必须保留。
124. 如果试验已设为 `招募中` 但尚未发布预筛表单，后台必须显著提示风险。
125. V1 不支持拖拽搭建器、多版本并行投放和复杂条件逻辑。

### 21. 线索池

126. 预筛提交后，系统必须在后台生成统一线索记录。
127. 线索池必须支持列表查看所有线索。
128. 线索池必须支持按试验、病种 / 适应症、城市、来源页面、状态、负责人、提交时间筛选。
129. 线索池默认按提交时间倒序排列。
130. 线索列表至少展示：
   - 患者姓名或匿名标识
   - 手机号脱敏展示
   - 试验名称
   - 病种 / 适应症
   - 来源页面
   - 提交时间
   - 当前状态
   - 负责人
131. 后台必须支持在线索列表中快速修改状态。
132. 后台应支持手动指定负责人。
133. V1 不要求自动分配负责人和复杂批量操作。

### 22. 线索详情与跟进

134. 线索详情页必须展示患者基础资料。
135. 线索详情页必须展示项目级预筛表单答案。
136. 线索详情页必须展示来源信息，至少包括：
   - 来源页面类型
   - 来源页面 ID
   - 若来自社区，则展示关联帖子 ID
137. 线索详情页必须支持以下状态流转：
   - 新线索
   - 待联系
   - 已联系
   - 跟进中
   - 关闭
138. 线索详情页必须支持录入跟进记录。
139. 跟进记录至少包含：
   - 跟进时间
   - 跟进方式
   - 跟进备注
   - 操作人
140. 跟进方式至少支持：
   - 电话
   - 微信
   - 短信
   - 其他人工沟通
141. 系统应支持录入下一次计划跟进时间。
142. 若同一患者对同一试验重复提交，后台必须可查看历史提交记录。
143. 若同一患者对不同试验提交，后台应可通过患者账号看到历史报名概况。
144. 线索详情页必须支持人工记录研究中心协同字段，至少包括：
   - 推荐中心
   - 中心联系人
   - 中心反馈状态
   - 中心反馈备注
145. V1 不建设研究中心独立登录页。

### 23. FAQ 管理

146. 后台必须支持 FAQ 列表页。
147. 后台必须支持新增、编辑、发布、下线 FAQ。

## Non-Functional Requirements

1. 性能：公开网站关键页面首屏 P95 响应时间必须小于 3 秒。
2. 性能：后台列表页在常见筛选条件下 P95 响应时间应小于 2 秒。
3. 稳定性：公开网站和后台月度可用性目标不低于 99.5%。
4. 安全：患者手机号等敏感信息在公开网站绝不展示，在后台列表页必须脱敏。
5. 安全：后台必须具备基础会话校验和角色权限校验。
6. 审计：试验、表单、线索、FAQ、社区审核、系统配置等关键动作必须可追溯。
7. 可维护性：前台与后台必须尽量复用统一组件与固定数据结构。
8. 可访问性：表单标签、错误提示和主要按钮必须清晰可读、可点击。
9. SEO：试验详情、FAQ 详情、帖子详情必须具备基础 SEO 友好 URL 和页面标题。
10. 合规：所有社区内容、FAQ 内容和试验文案都必须支持人工下线和审计追溯。

## QA & Testing Impact

### High-risk surfaces

1. 试验公开状态与前台可见性的同步。
2. 项目级预筛表单配置与前台渲染的一致性。
3. 登录回跳、资料预填和我的报名数据一致性。
4. 社区审核状态与前台帖子、评论展示一致性。
5. 社区与试验详情中的预筛入口是否始终指向当前有效表单。

### New test coverage required

1. 公开站桌面端与移动端响应式适配。
2. 患者登录前后触发预筛的差异流程。
3. 不同字段类型的表单渲染与校验。
4. 同一患者重复提交、跨项目提交场景。
5. 帖子匿名与实名发布的展示差异。
6. 社区帖子和评论的通过、驳回、隐藏、精选分支。
7. `operator` 与 `admin` 的基础权限边界。
8. 传统病种与医美适应症共用标签体系时的筛选与展示。

### Regression scope

1. 首页、试验列表、试验详情。
2. 预筛表单、提交成功页、我的报名。
3. FAQ 列表与详情。
4. 社区首页、病种社区、帖子详情、评论区。
5. 后台试验管理、线索池、内容审核。

### Edge cases to verify

1. 试验已发布但表单被下线后，前台 CTA 如何处理。
2. 同一患者短时间重复提交同一项目时，线索和报名记录如何保留。
3. 精选帖子随后被隐藏后，前台推荐位是否同步下线。
4. 用户未登录点击社区预筛入口时，登录后的回跳是否正确。
5. 医美项目和传统病种项目共用筛选器时，结果是否正确。

## V1 Implementation Boundaries

1. 只做响应式 Web 网站，不做 App 和完整小程序。
2. 只做统一运营后台，不做研究中心后台。

7. 统计只做固定指标卡和少量趋势图，不做分析平台。
8. 研究中心协同先通过线索详情人工字段承接。

## Build-Friendly Assumptions

1. 前台和后台共用一套代码仓与一套数据库。
2. URL、页面、状态机、字段类型尽量固定，不在开发中引入自由扩展机制。
3. 先由后台人工配置首页推荐位、试验推荐位、社区推荐位和 FAQ 推荐位。
4. 社区内容默认走基础审核，不引入复杂机器审核流。
5. 联系方式、免责声明、隐私政策等可以先使用固定内容模板。
6. 登录能力优先满足流程闭环，不在首版追求复杂账号安全体系。
7. 任何可能导致系统复杂度指数上升的需求，默认放到 V2 讨论。

## 前端实现约束（Frontend Implementation Constraints）

本节为 V1 必须遵守的前端实现约束，作为 Web Interface Guidelines 的项目落地版本。所有页面与组件实现必须满足本节，研发与 vibe coding 输出不得自由发挥。

### F1. 设计令牌与主题

1. V1 仅交付浅色主题。`<html>` 必须设置 `color-scheme: light`，并配置 `<meta name="theme-color">` 与背景一致。
2. 所有色彩、间距、圆角、字号统一通过 CSS 变量（design tokens）管理，禁止硬编码。
3. 主品牌色用于主 CTA 与重点强调；中性灰用于正文、边框、卡片背景；语义色覆盖成功、警告、危险、信息四类。
4. 字体优先级：`-apple-system, BlinkMacSystemFont, "Segoe UI", "Microsoft YaHei", "PingFang SC", sans-serif`。
5. 数字列必须使用 `font-variant-numeric: tabular-nums`。
6. 标题区使用 `text-wrap: balance`。

### F2. 文案与排版规范

1. 省略号统一使用 `…`（U+2026），禁止使用 `...`。
2. 引号统一使用弯引号 `""` `''`，禁止使用直引号。
3. 全部以 `/` 表示并列关系时统一为"前后无空格"形式（如 `登录/注册`），仅在病种/适应症等长名称并列时允许 `空格 / 空格` 形式，但全文必须统一一种风格，不混用。
4. 加载中文案必须以 `…` 结尾（如"加载中…"）。
5. 主 CTA 必须使用动词短语，禁止使用"确定/提交"等弱标签（如使用"立即预筛"而非"提交"）。
6. 所有错误提示必须包含修复建议，不得仅写"输入错误"。
7. 数字、日期、相对时间统一使用 `Intl.NumberFormat` 与 `Intl.DateTimeFormat`，不得手写格式。
8. 品牌名与英文产品名加 `translate="no"`。

### F3. 表单与输入

1. 所有 `<input>`、`<select>`、`<textarea>` 必须有显式 `<label htmlFor>` 或包裹式 label，不允许只用 `placeholder` 替代标签。
2. 手机号字段必须使用 `type="tel" inputmode="tel" autocomplete="tel"`。
3. 短信验证码字段必须使用 `inputmode="numeric" autocomplete="one-time-code"`。
4. 姓名/性别/年龄/城市等基本资料字段必须配置正确的 `autocomplete`（`name`、`sex`、`bday`、`address-level2` 等）。
5. 数字字段必须使用 `inputmode="numeric"` 与 `pattern`。
6. 邮件、账号、验证码字段必须 `spellcheck="false"`。
7. 占位符必须以 `…` 结尾或给出示例值（如"如：13800000000"）。
8. 禁止 `onPaste preventDefault`，手机号/验证码字段必须支持粘贴。
9. 提交按钮在请求开始前保持可点击；请求中显示 spinner、禁用并阻止重复提交。
10. 校验失败时必须在字段下方内联展示错误，并自动聚焦首个错误字段。
11. 已自动预填的字段必须有视觉提示（如灰底+"已自动带入，可修改"提示），允许用户修改。
12. 表单页有未保存修改时，离开必须二次确认。

### F4. 可访问性最低标准

1. 全站必须达到 WCAG 2.1 AA 等级。
2. 所有交互元素必须键盘可达，焦点必须可见（使用 `:focus-visible` + 明显焦点环），禁止 `outline: none` 而无替代焦点样式。
3. 图标按钮必须有 `aria-label`，装饰性图标必须 `aria-hidden="true"`。
4. 异步反馈（吐司、校验提示、加载状态）必须使用 `aria-live="polite"`。
5. 标题层级必须严格递进（h1 → h2 → h3），不得跳级。
6. 公开网站每个页面必须提供"跳转到主内容"的 skip link。
7. 移动端可点击区域最小 44×44 px。
8. 禁止 `user-scalable=no` 与 `maximum-scale=1`。
9. `<a>` 用于导航，`<button>` 用于动作，禁止 `<div onClick>`。
10. 所有 `<img>` 必须有 `width`、`height`、`alt`，装饰图 `alt=""`。

### F5. URL 与状态

1. 公开网站列表/详情/筛选状态必须可深链：
   - 试验列表筛选条件（病种、城市、招募状态、关键字）写入 URL query。
   - 基础统计页时间范围、维度切换写入 URL query。
   - 社区分区切换写入 URL path 或 query。
2. 所有内部跳转必须使用 `<a href>` 或框架等价 `<Link>`，禁止 `onClick + window.location`。
3. 来源归因参数（`utm_source`、`source_post_id`、`source_page_type`）必须从 URL 透传到预筛提交埋点。
4. 试验详情、FAQ 详情、帖子详情必须配置 `<title>`、`<meta name="description">`、`<link rel="canonical">`、Open Graph 与 Twitter Card。
5. 试验详情页应实现 Schema.org `MedicalStudy` 结构化数据，提升 SEO。

### F6. 列表性能与空状态

1. 任何列表渲染条目超过 50 必须虚拟化（含线索池、社区帖子列表、审核队列、我的报名）。
2. 列表必须实现服务端分页或无限滚动，单次请求默认页大小 20。
3. 所有列表页必须设计加载中（骨架屏）、空状态、错误状态三类展示，不得只画"有数据"形态。
4. 空状态必须给出"下一步操作"引导，不得只显示"暂无数据"。
5. 列表数据列必须设置 `min-w-0` 与 `truncate`/`line-clamp` 防止撑破布局。

### F6. 隐私与脱敏

1. 患者手机号在公开网站任何页面禁止展示。
2. 患者手机号在后台所有列表页必须脱敏（如 `138****0000`）。
3. 后台线索详情页默认脱敏，运营人员主动点击"查看完整手机号"按钮才解密展示，且每次点击必须写入审计日志。
4. 公开网站 URL 中禁止暴露 `patient_user_id`、`lead_id` 等可枚举内部 ID，对外使用短码或 UUID。
5. 报名记录、社区"我的"页只能查看本人数据，必须服务端校验登录态与所有权。

### F7. 破坏性动作与确认

1. 以下动作必须二次确认对话框，并明确告知影响：
   - 试验下线（提示：将关闭对应预筛入口）
   - 预筛表单废止
   - 后台账号停用、密码重置
   - 社区帖子/评论"驳回""隐藏"
   - 用户删除自己的帖子或评论
2. 后台审核驳回必须强制填写驳回原因，且原因展示在审计日志中。
3. 所有提交类操作成功后必须使用 `aria-live="polite"` 反馈结果。

### F8. 动效与暗色模式

1. 所有动效必须遵循 `@media (prefers-reduced-motion: reduce)` 关闭动画。
2. 仅允许动 `transform` 与 `opacity`，禁止 `transition: all`。
3. SVG 动效作用于 `<g>` 包裹层，并设置正确 `transform-origin`。
4. V1 不实现暗色模式；如未来 V2 实现，必须在 `<html>` 添加 `color-scheme: dark`，并保证 `<meta name="theme-color">` 同步切换。

### F9. 水合与受控组件

1. React/Vue 受控输入必须 `value` + `onChange` 配对，禁止只设 `value` 不设 `onChange`。
2. 服务端渲染时间、相对时间、随机内容必须在客户端二次水合，或对该节点使用 `suppressHydrationWarning`，避免水合 mismatch。
3. 日期相关展示必须在用户时区下渲染，禁止以服务端时区作为最终展示。

### F10. 图片与媒体

1. 所有 `<img>` 必须显式 `width`、`height` 防止累积布局位移。
2. 首屏关键图片使用 `fetchpriority="high"`，其余 `loading="lazy"`。
3. 患者头像/精选帖子封面必须配置兜底占位图。
4. CDN 域名必须配置 `<link rel="preconnect">`，关键字体配置 `<link rel="preload">`。

### F11. 触控与移动端

1. 所有可点击元素设置 `touch-action: manipulation`。
2. `-webkit-tap-highlight-color` 必须显式设置（默认透明或品牌淡色）。
3. 模态框/弹层设置 `overscroll-behavior: contain`。
4. 移动端表单字段必须避免被键盘遮挡（聚焦时滚动入视口）。

### F12. 一致性与组件库

1. 全站必须使用统一组件库实现按钮、输入、卡片、对话框、Toast、Tag、状态徽章、空状态、骨架屏。
2. 状态徽章颜色必须与状态机枚举一一映射，前后台共用同一映射表。
3. 不允许业务页面自行定义新颜色或新尺寸，必须复用 design tokens。

### F13. 验收清单（DoD 增项）

1. Lighthouse 移动端评分：性能 ≥ 80、可访问性 ≥ 95、最佳实践 ≥ 90、SEO ≥ 90。
2. 主链路（首页 → 试验列表 → 试验详情 → 预筛 → 提交成功）必须通过键盘完整走通。
3. 主链路必须通过 VoiceOver/NVDA 朗读校验，不得出现"button button"等冗余朗读。
4. 所有表单字段必须通过浏览器自带"自动填充"成功填入正确字段。
5. 所有列表页必须在 1000 条数据下保持滚动 60fps。

## 设计语言（Design Language Reference）

参考 SubjectWell（https://subjectwell.com/）的患者侧设计原则，结合本项目实际，V1 视觉语言收敛如下：

### D1. 患者侧（公开网站）

1. 视觉风格：明亮、温暖、医疗专业，大量留白，圆角柔和。
2. 主色：医疗蓝（信任）+ 暖橙辅色（行动 CTA），背景以白与浅灰为主。
3. 首页结构对齐 SubjectWell：Hero（主张+双 CTA+患者意象图）→ 平台价值三要素 → 重点病种入口 → 推荐试验 → 流程说明 → 数据墙（可信度）→ 信任层（FAQ + 社区）→ 页脚。
4. 患者照片必须体现多元化（年龄、性别、病种），优先采用真实生活场景。
5. 文案风格：第二人称、短句、动词起头、避免监管术语。

### D2. 后台侧（统一运营后台）

1. 视觉风格：高密度、信息为先，弱化装饰，强化数据可读。
2. 配色保留浅色主题，状态徽章四色（success/warning/danger/info）。
3. 页面统一三栏：左侧固定导航 + 顶部全局工具条 + 主区域。
4. 关键运营动作（新建试验、新增线索处理、审核）必须在 1 屏内完成。

### D3. 跨端共享

1. 公开网站与后台共用 design tokens、按钮、输入、表格、对话框组件。
2. 状态机文案与颜色映射统一定义，避免前后台不一致。
3. 公开网站移动端优先（mobile-first），后台仅保证桌面端体验。
