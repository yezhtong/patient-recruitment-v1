## 文档摘要

本 PRD 面向“临床试验患者招募网站”的项目级预筛表单能力 V1。

目标不是做一个通用低代码表单平台，而是：

1. 让不同项目可以配置少量差异化问题
2. 让患者在统一体验下完成预筛提交
3. 让已登录患者复用基础资料，提高提交效率
4. 让运营在后台管理项目、表单和线索形成闭环

本版特别强调两点：

1. 需要同时适配传统病种项目与医美相关适应症项目
2. 需要控制复杂度，适合短周期 `vibe coding`

---

## Problem Statement

当前患者招募网站若只使用一套完全固定的预筛问卷，会遇到以下问题：

1. 不同临床试验的纳入排除标准不同，固定表单无法覆盖项目差异。
2. 肿瘤、慢病、皮肤科、医美相关适应症在前置筛查问题上差异明显，同一份表单容易过度冗长或信息不足。
3. 如果每个项目都由研发手工写死表单，维护成本高，上线速度慢，不适合持续上新项目。
4. 若直接做成通用低代码表单引擎，又会显著增加开发复杂度、渲染复杂度和测试成本，不利于 V1 落地。
5. 患者如果每次报名都要重复填写基础信息，提交体验差，转化会下降。

因此，平台需要一个“可配置但受控”的项目级预筛表单能力。

---

## North Star Connection

平台级 North Star：`Weekly Qualified Prescreen Submissions`

项目级预筛表单对 North Star 的贡献路径：

1. 让不同项目展示更准确的问题，提高有效提交率。
2. 让已登录患者自动复用基础资料，提高表单完成率。
3. 让运营可以更快上新项目，而不需要频繁依赖研发改前台页面。
4. 让线索详情结构更统一，便于运营判断和后续跟进。

模块级核心假设：

- 若项目问题能够更贴近项目实际条件，同时保留统一的提交流程，则预筛启动到提交率会高于完全固定问卷。

## Target Users

### 主要用户

1. 注册患者/家属
   目标：快速完成某个项目的资格预筛，尽量少重复填写资料。

### 次要用户

1. 匿名访客
   目标：浏览项目并尝试进入预筛，但最终提交前建议转为登录用户。
2. 运营用户 `operator`
   目标：为每个项目配置合适的预筛问题，并查看提交结果。
3. 管理员 `admin`
   目标：维护字段组件和系统级配置。

---

## Scope

### In Scope

1. 每个试验项目拥有独立的预筛表单配置。
2. 表单由“基础资料字段 + 项目特有字段”组成。
3. 已登录患者自动预填基础资料字段。
4. 有限字段类型和有限字段规则。
5. 统一的公开端表单渲染。
6. 表单提交后生成结构化线索。
7. 后台可配置、预览、发布项目表单。

### Out of Scope

1. 通用低代码表单引擎。
2. 可视化拖拽搭建器。
3. 多层复杂条件跳转。
4. 文件上传字段。
5. 表单 A/B Test。
6. 多版本并行投放。

---

## User Stories

1. 作为患者，我希望不同项目的问题更贴近该项目本身，这样我不会被无关问题打断。
2. 作为患者，我希望登录后能自动带出基础资料，这样我不用每次重复填写。
3. 作为患者，我希望表单问题尽量清晰易懂，这样我知道自己是否值得继续报名。
4. 作为运营，我希望为某个项目配置专属预筛问题，这样我能更准确收集线索。
5. 作为运营，我希望上线新项目时不需要研发手动改页面，这样我能更快发布项目。
6. 作为运营，我希望在后台看到结构化的表单答案，这样我能快速判断线索价值。

## Functional Requirements

### 1. 表单模型

1. 系统必须支持“一个项目对应一个生效中的预筛表单”。
2. 预筛表单必须拆分为两层：
   - 基础资料字段
   - 项目特有字段
3. 基础资料字段必须可复用于多个项目。
4. 项目特有字段必须由项目配置决定是否展示。
5. 表单配置保存后必须支持“预览”和“发布”两种状态。

### 2. 支持的字段类型

6. V1 必须支持以下字段类型：
   - 单行文本
   - 多行文本
   - 单选
   - 多选
   - 是/否
   - 数字
   - 日期
7. V1 不支持文件上传字段。
8. V1 不支持富文本回答字段。

### 3. 字段属性

9. 每个字段必须支持以下基础属性：
   - 字段编码
   - 字段标题
   - 字段说明
   - 字段类型
   - 是否必填
   - 排序顺序
10. 选择类字段必须支持选项列表配置。
11. 数字字段应该支持最小值和最大值配置。
12. 日期字段应该支持日期范围配置。
13. 字段必须支持标记为“基础资料字段”或“项目字段”。

### 4. 受控规则能力

14. V1 必须支持基础必填校验。
15. V1 必须支持手机号格式校验。
16. V1 应支持简单提示规则：
   - 若用户答案明显不符合项目核心要求，展示提示文案
17. V1 不支持多层条件跳转。
18. V1 不支持跨字段复杂逻辑计算。
19. V1 不支持字段之间的动态联动字典。

### 5. 患者端渲染与交互

20. 患者进入项目预筛页时，系统必须根据该项目加载对应表单配置。
21. 对已登录患者，系统必须自动预填已保存的基础资料字段。
22. 患者端必须明显区分“基础资料”和“项目相关问题”。
23. 每个字段必须支持展示标题和说明文案。
24. 患者提交前必须完成必填校验。
25. 提交成功后必须进入统一成功页。

### 6. 基础资料字段建议

26. V1 默认基础资料字段建议包括：
   - 姓名
   - 性别
   - 年龄
   - 手机号
   - 所在城市
   - 主要疾病/适应症
27. 基础资料字段必须可同步到 `patient_profile`。
28. 基础资料字段不应被项目方随意删除，但可配置是否在当前项目中显示或只读展示。

### 7. 项目特有字段示例

29. 针对传统病种项目，可配置如下问题：
   - 是否已确诊某疾病
   - 当前治疗阶段
   - 是否做过相关检查
   - 既往治疗方式
30. 针对医美相关适应症项目，可配置如下问题：
   - 是否存在某类皮肤困扰
   - 困扰持续时间
   - 是否接受过相关治疗
   - 近 6 个月是否进行过某类医美项目
31. 所有项目字段都必须使用统一字段组件库，不允许自定义前端组件。

### 8. 后台配置能力

32. 运营后台必须支持为项目创建和编辑预筛表单。
33. 运营后台必须支持从系统字段组件库中添加字段。
34. 运营后台必须支持设置字段顺序。
35. 运营后台必须支持设置字段是否必填。
36. 运营后台必须支持设置字段说明文字和选项。
37. 运营后台必须支持表单预览。
38. 运营后台必须支持表单发布。
39. V1 不要求字段模板市场或表单复制库，但应允许复制已有项目表单作为新项目起点。

### 9. 提交与线索生成

40. 患者提交表单后，系统必须生成一条 `prescreen_submission` 记录。
41. 系统必须同步生成或更新对应 `lead`。
42. `lead` 必须与项目、患者账户、来源页面和表单答案关联。
43. 运营在线索详情页必须能够查看：
   - 基础资料
   - 项目特有答案
   - 提交时间
   - 来源页面
44. 若同一患者对同一项目重复提交，系统应提示已提交并记录最新提交。

### 10. 版本与变更控制

45. V1 必须保证任一时间点只有一个“当前生效表单版本”。
46. 若运营修改并重新发布表单，新提交使用新版本，旧提交保留历史答案。
47. 旧版本答案必须可在后台查看。
48. V1 不要求支持同时多版本并行投放。

### 11. 表单与社区联动

49. 社区帖子详情页中的“立即预筛”必须跳转到对应项目当前生效表单。
50. 试验详情页中的“立即预筛”必须跳转到对应项目当前生效表单。
51. 提交成功页必须可引导返回相关 FAQ 或病种 / 适应症社区。

---

## Non-Functional Requirements

1. 性能：预筛表单页首屏加载 P95 必须小于 3 秒。
2. 稳定性：提交接口必须具备防重复提交机制。
3. 安全：表单提交涉及手机号等敏感信息，必须符合平台隐私要求。
4. 审计：后台对表单配置的修改和发布动作必须保留日志。
5. 可用性：字段标签、说明和错误提示必须清晰可见。
6. 可维护性：前端渲染必须基于受控字段组件库，而非为单项目写定制页面。

## Data Model Impact

### 核心实体建议

1. `prescreen_form`
   - id
   - trial_id
   - status
   - version
   - published_at
   - created_by

2. `prescreen_form_field`
   - id
   - form_id
   - field_key
   - field_label
   - field_type
   - field_help_text
   - is_required
   - sort_order
   - options_json
   - min_value
   - max_value
   - date_min
   - date_max
   - is_profile_field

3. `prescreen_submission`
   - id
   - form_id
   - form_version
   - trial_id
   - patient_user_id
   - source_page
   - submitted_at
   - status

4. `prescreen_answer`
   - id
   - submission_id
   - field_id
   - field_key
   - answer_value

5. `patient_profile`
   - patient_user_id
   - name
   - gender
   - age
   - phone
   - city
   - primary_condition

6. `lead`
   - id
   - trial_id
   - patient_user_id
   - latest_submission_id
   - status
   - source_page

### 关系说明

1. 一个 `trial` 对应一个当前生效中的 `prescreen_form`。
2. 一个 `prescreen_form` 对应多个 `prescreen_form_field`。
3. 一个 `prescreen_submission` 对应多条 `prescreen_answer`。
4. `patient_profile` 与 `prescreen_form_field` 中的基础资料字段共享。

## API Changes

### 公开网站接口

1. `GET /trials/{id}/prescreen-form`
   返回当前项目的表单定义。
2. `POST /trials/{id}/prescreen-submissions`
   提交预筛答案。
3. `GET /me/profile`
   获取患者基础资料。
4. `PATCH /me/profile`
   更新患者基础资料。
5. `GET /me/prescreen-submissions`
   获取我的报名记录。

### 运营后台接口

6. `GET /admin/trials/{id}/prescreen-form`
7. `POST /admin/trials/{id}/prescreen-form`
8. `PATCH /admin/prescreen-forms/{id}`
9. `POST /admin/prescreen-forms/{id}/publish`
10. `GET /admin/leads/{id}`

## QA & Testing Impact

### High-risk surfaces

1. 表单配置页与前台渲染的一致性。
2. 登录后基础资料预填逻辑。
3. 同一项目修改表单后，历史提交记录的可追溯性。
4. 社区 / 试验详情页跳转到当前生效表单的准确性。

### New test coverage required

1. 不同字段类型的渲染与提交。
2. 必填校验、手机号校验、数字/日期边界校验。
3. 已登录与未登录用户的提交流程差异。
4. 表单发布前后版本切换。
5. 传统病种项目与医美适应症项目的典型题型覆盖。

### Edge cases to verify

1. 患者在填写过程中登录状态失效后如何处理。
2. 字段被运营删除后，历史答案如何展示。
3. 患者重复提交同一项目时如何提示。
4. 项目下线后预筛入口是否关闭。

## V1 实现边界建议

1. 一个项目只允许一个当前生效表单。
2. 字段组件固定在 7 类以内。
3. 字段规则只支持必填、选项和简单边界校验。
4. 表单逻辑不做复杂条件跳转。
5. 患者基础资料与项目字段分层存储和渲染。
6. 表单配置优先用结构化表 + JSON 辅助存储，而不是做完整低代码平台。
