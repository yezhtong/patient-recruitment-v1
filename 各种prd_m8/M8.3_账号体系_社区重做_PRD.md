## 二、用户故事

### 游客
- 作为游客，我能看到社区时间流（公开帖子），但发帖/评论/加入社区**必须先注册**

### 注册用户（patient）
- 作为一个慢性病患者，注册时能**自由描述症状**（"头胀、早上手麻、失眠"），提交后系统用 **AI（DeepSeek）分析**给出**标准疾病名 + 建议标签**（如"血压偏高相关 · 神经系统症状"），我可以确认或删除不准的标签，系统记住，**下次匹配试验更准**
- 作为患者，我可以**选择**（不强制）上传确诊书/检查报告（图片或 PDF），运营看到后帮我更精准匹配
- 作为患者，我点「试验匹配助手」能**对话式问 3-5 个问题**后得到一个**固定格式**的推荐：
  - 我的症状可能方向（示例：提示关注心血管系统）
  - 建议就医科室（示例：心内科）
  - 当前平台可参加的试验 3 条（带链接）
- 作为患者，我看到匹配助手**反复说明**它不是医生、不能诊断、所有信息仅供参考
- 作为患者，我可以加入最多 3 个社区（按病种/话题/身份分）；加入前系统会**要求我补完症状**（否则无法加入）
- 作为患者，我在社区发帖，帖子会**先经过 AI 审核**，轻微内容秒过，可疑内容转人工；通过后出现在时间流

### 医生（role=doctor，后台手动分配）
- 作为医生，我的帖子和评论旁显示「✓ 医生」tag（绿色徽章）
- 作为医生，我**不受**反爬封号限制（M8.1 已预留）
- 作为医生，我**不能**自己申请成为医生，必须由 admin 分配

### AI 账号（系统账号，`role=ai`）
- AI 账号**不代表真人**，显示「🤖 AI 助手」tag（灰色徽章 + 特殊背景）
- 触发时机：
  - 新注册用户首次进入社区 → AI 账号在"新手引导"标签流下自动发一条欢迎贴 + 引导链接
  - 用户发帖命中"常见问题模板"（如"怎么报名""多久回复我"）→ AI 账号自动评论一条 FAQ 引用
- AI 账号的发帖/评论走**特殊通道**（无审核，但每条都打 ai-generated 标记便于追溯）

### admin / operator
- 作为 admin，我可以在 `/admin/users` 给某用户分配 `role=doctor`（操作写入审计）
- 作为 admin，我可以在 `/admin/users` 查看用户的**症状记录**和**上传的诊断文件**
- 作为 operator，我在社区审核队列看到每条待审内容旁边有**AI 判断 + 置信度**（通过/拒绝/建议人工），我可以一键认同 AI 结论，也可以填写**覆盖理由**做人工决策
- 作为 admin，我可以在 `/admin/llm-logs` 看到所有 AI 审核 / 匹配助手 / AI 回复的调用记录（复用 M8.2 表）

## 五、业务规则详解

### 5.1 注册流症状填写（/auth/register/symptoms）

- 步骤编号：step 2（step 1 手机验证 → step 2 症状 → step 3 可选上传）
- **交互流程**（决策 7 反转后）：
  1. **输入态**：自由文本 textarea（10-500 字），占位符"比如：最近半年血压偏高，经常头晕，早上手麻。"
  2. 用户点"AI 分析"按钮 → **loading 态**（约 3-5s，展示"AI 正在分析你的症状…"）
  3. 后端调 `src/lib/disease-matcher.ts` → DeepSeek 返回 `[{label, keyword, confidence}]` 数组
  4. **结果态**：上方显示用户原文，下方展示 AI 返回的**标签 chips**（最多 8 条）+ 每个 chip 旁有 `×` 删除按钮
  5. 用户可点 `×` 删除不准的标签，也可点"重新分析"重新调
- **跳过**按钮：显著标注"跳过将影响后续试验匹配准确度"（不强制）
- **提交**（点"下一步"）：
  - 更新 `User.symptomsText` / `User.aiDiseaseTags` / `symptomsUpdatedAt`
  - 跳转 step 3（上传）或 "推荐分区"弹窗（§5.5.1 / §5.6）

### 5.2 疾病匹配服务（`disease-matcher.ts`）

> **决策 7 反转（2026-04-22）**：**废止 ICD-10 本地词典**。改用 DeepSeek 动态分析，无需建本地库、无需导入 2 万条数据、无需 `seed-icd10.ts`。

- 文件位置：`app/src/lib/disease-matcher.ts`
- 核心函数：
  ```ts
  export interface DiseaseTag {
    label: string;       // 中文疾病名，如 "高血压"
    keyword: string;     // 英文短关键词（用于后续匹配分区），如 "hypertension"
    confidence: number;  // 0..1
  }
  export async function analyzeSymptoms(freeText: string): Promise<DiseaseTag[]>
  export async function recommendGroups(userTags: DiseaseTag[], groups: CommunityGroup[]): Promise<{group, matchScore, matchedKeywords}[]>
  ```
- 内部走 `LlmProvider`（M8.2 已做的 DeepSeek 抽象），**未配置 DEEPSEEK_API_KEY 时自动 Mock** 返回 3 条 fixture 数据
- **prompt 模板（system 部分）大意**：
  > 你是医疗领域的疾病识别助手。用户描述了自己的症状，请返回 JSON 数组，每项含 `label`（中文疾病名）/ `keyword`（英文短关键词）/ `confidence`（0-1 浮点）。
  > 限制：1) 最多返回 8 条；2) 只返回中文常见疾病，不要用 ICD 编码；3) 严禁给出治疗或用药建议。
  > 输出格式示例：`[{"label":"原发性高血压","keyword":"hypertension","confidence":0.92}]`
- **所有调用写 `LlmCallLog`**（scenario=`disease_analysis`），沿用 M8.2 的日志后台
- 缓存：同一文本 hash 60s 内命中返回上次结果（避免重复调用）

### 5.3 试验匹配助手（/match-assistant）

- **免责声明横幅**（页面顶部 + 每条响应底部）：
  > ⚠️ 本助手不是医生，不提供诊断或治疗建议。所有信息仅供参考，具体医疗决定请咨询专业医师。如遇紧急情况请拨打 120 或前往医院急诊。

- **交互形式**：对话式（比表单更自然，用户留存更高）
  - 初始问候："你好，告诉我你最近的主要不适，我帮你看看可以参加哪些试验。（不是诊断）"
  - 用户输入 → 发给 DeepSeek
  - **prompt 模板**（system）：
    > 你是一个临床试验推荐助手。严格遵守：
    > 1. **禁止**给出医学诊断、处方、治疗建议
    > 2. **禁止**说"你可能得了 XX 病"或"建议服用 XX"
    > 3. **必须**按以下 JSON 格式输出：
    >    `{ "symptomDirection": "...", "suggestedDepartment": "...", "followUpQuestions": ["..."], "readyToRecommend": false }`
    > 4. 当 readyToRecommend=true 时表示信息足够，可以推荐试验
    > 5. symptomDirection 只说"提示关注 XX 系统"（如"心血管""内分泌""神经"），**不说病名**
    > 6. suggestedDepartment 只说科室名称（如"心内科""神经内科"）
  - 后端根据 `readyToRecommend`：
    - false → 继续问下一题
    - true → 取 `symptomDirection` 关键词 + 用户 `aiDiseaseTags` 里的 `keyword` 数组 → 查 `ClinicalTrial where disease LIKE %keyword% OR title LIKE %keyword%` 返回 top 3（**决策 7 反转**：不再查 `recommendedIcdCodes`，改为按疾病名/标题的自然语言模糊匹配，辅以 DeepSeek 二次排序）
  - 输出卡片（固定模板，UI 严格不变）：
    ```
    [红底] ⚠️ 免责声明横幅

    1. 你描述的症状可能关注方向
       {symptomDirection}

    2. 建议就医科室
       {suggestedDepartment}（本平台**不代替医生面诊**，建议先到该科室就诊）

    3. 目前可以参加的临床试验（{count} 条）
       - [试验标题 1]（{city} · {phase}）[查看详情]
       - [试验标题 2] ...
       - [试验标题 3] ...

    [再次重复免责]
    ```
- 会话：写 `MatchAssistantSession`，方便后续复盘 + admin 抽查
- 消息上限：5 轮对话必出结果（避免无限 LLM 成本）
- 输出后下方按钮：「参加试验」（跳详情）/「继续聊」（重置会话但保留症状）/「导出给医生看」（生成 PDF 的 V2 需求，这次不做）

### 5.4 确诊记录上传（/auth/register/records 或 /me 里后补）

- 字段：
  - 文件（image/*, application/pdf，单文件 ≤ 5MB，最多 5 个）
  - 可选备注（"这是 2024 年 3 月体检报告"）
- 存盘：`public/uploads/medical-records/{cuid}.{ext}`
  - **路径对外不可直接访问**：通过 `/api/user/records/[id]` 鉴权后返回
  - 管理端通过 `/admin/users/[id]` 的签名 URL 查看
- **合规要点**：
  - 上传前弹确认："上传的医疗信息仅用于试验匹配，九泰临研将严格保密，不会对外分享"
  - 用户可以在 `/me` 随时删除已上传文件（软删除 → 文件保留 30 天可恢复）
- 用户不上传不影响注册（可跳过）

### 5.5 社区首页 v2（保留病种分区）

> **2026-04-22 路径反转**：原设计"去分区 + 时间流 + 标签三类"已废止；保留 M5 的 6 个病种分区，首页改成"我加入的 / 推荐加入 / 其他"三段布局。

- `/community` 新布局（**三段式**，自上而下）：
  1. **「我加入的分区」chip 条**（最多 3 个）：
     - 每个 chip 显示：分区色 + 名称 + 未读帖子数（可选徽章）
     - 点击进入 `/community/groups/[slug]` 沿用 M5 分区详情
     - 未加入任何分区时显示占位："还没加入任何分区，看看下面的推荐？"
  2. **「推荐加入」段**（基于用户 `aiDiseaseTags` 的 AI 语义匹配，最多 3 个卡片）：
     - 推荐算法见 §5.5.1；无匹配时隐藏整段
     - 每个卡片："分区色块封面 + 分区名 + 简介 + 一句匹配理由（"因为你填了血压偏高相关症状"）+「加入」按钮"

  3. **「其他可加入分区」段**：
     - 展示剩余未加入、未被推荐的分区卡片（按 `sortOrder`）

     - 点"加入"→ 同样触发 §5.6 弹窗
- 顶部右侧固定按钮：「发帖」→ 弹出"选择要发到哪个分区"（**仅列出"我加入的分区"**；未加入任何分区时 disabled + 提示"先加入一个分区再发帖"）
- **空状态**：

  - 已加入满 3 个："你已加入 3 个分区（上限），想加入新分区请先在「我的 · 我的分区」退出一个。"

#### 5.5.1 推荐分区算法（决策 7 反转 · AI 语义匹配）

- **输入**：`User.aiDiseaseTags`（数组：`[{label,keyword,confidence}]`）+ 所有启用分区的 `CommunityGroup`
- **候选过滤**：先按 `sortOrder` 取启用的分区，排除用户已加入（`leftAt=null`）的
- **匹配方式**（两档，按成本选）：
  1. **轻量档（默认）**：纯**字符串包含**——`user.aiDiseaseTags[].keyword` 或 `.label` 与 `group.recommendedDiseaseKeywords[]` 或 `group.diseaseTag` 做不区分大小写的子串匹配。命中数越多 `matchScore` 越高
  2. **重档（可选 · 卡顿时降级）**：调 `disease-matcher.ts` 的 `recommendGroups(userTags, groups)` 走 DeepSeek 做语义匹配（如"高血压"↔"心血管疾病"也能命中）；单次调用约 ¥0.005，新用户登录首次推荐时值得
- 取 `matchScore desc` 前 3 个
- **降级**：用户未填任何 `aiDiseaseTags`（注册时跳过了 AI 分析）时，整段"推荐加入"**不显示**，用户自行从"其他可加入分区"手动选

> 备注：两档匹配由 `src/lib/disease-matcher.ts` 统一暴露，调用方传入 `mode="string" | "semantic"`，默认 `"string"`。

### 5.6 加入分区 + 补症状 + 回写账号闭环

**入口**（两个）：
- 注册流 `/auth/register/symptoms` 完成后 → **推荐分区弹窗**（见 §5.5 推荐算法），用户可勾选加入
- `/community` 首页点"推荐加入"或"其他分区"卡片上的「加入」按钮 → **加入前补症状弹窗**

**流程**（两个入口走同一段逻辑）：

1. **前置校验**
   - 未登录 → 跳 `/auth`
   - 当前 `membership.leftAt=null` 数量 ≥ 3 → 阻断提示"你最多加入 3 个分区（上限），请先在「我的」退出一个"
   - 已加入此分区 → 直接跳 `/community/groups/[slug]`（幂等）

2. **弹窗表单**（标题"加入【{groupName}】前，请补充您的疾病信息"）—— 决策 7 反转后的**两态流程**：

   **态 1 · 输入态**：
   - 字段：
     - **自由文本** `joinSymptoms`（必填，10-200 字）：占位符"例如：2024 年确诊高血压，目前吃络活喜控制"
     - 同意勾选："我知晓本分区讨论内容不构成医疗建议"（必勾，合规护栏）
   - CTA：「提交并让 AI 分析」
   - **不再有 ICD-10 chips 预选**（决策 7 反转删除）

   **态 2 · AI 分析 → 确认态**（提交后展示，约 3-5s loading 后到位）：
   - server 调 `disease-matcher.ts` · `analyzeSymptoms(joinSymptoms)` → 返回 `DiseaseTag[]`
   - 前端展示"AI 分析出来的标签"（每个 chip 旁带 × 可删）+ 提示"如果有不准的，点 × 删掉"
   - CTA：「确认加入」→ 提交

   **最终提交**（态 2 的确认）：
   1. 写 `UserGroupMembership` 记录：`userId / groupId / joinedAt=now() / joinSymptoms / joinDiseaseTagsJson`（后者存 AI 分析 + 用户手删后的最终 `DiseaseTag[]`）
   2. **回写账号**（累加，不覆盖）：
      - `User.symptomsText`：如果为空则 = `joinSymptoms`；否则追加 `"\n---\n[加入{groupName}时补充] {joinSymptoms}"`
      - `User.aiDiseaseTags`：现有 `DiseaseTag[]` ∪ 本次确认后的 tags（按 `keyword` 去重，保留 confidence 最高那条）
      - `User.symptomsUpdatedAt = now()`
   3. 跳转 `/community/groups/[slug]`

3. **为什么"每次加入新分区都要补"**
   - 每个病种关心的细节不同（糖友关心血糖数值，乳腺癌患者关心术后分期），一次注册填的症状无法精准覆盖各分区
   - 多次补充 + 回写账号可让"我的症状"随时间增厚，为后续**试验匹配助手**提供更全数据
   - 对用户而言摩擦小（每次只补一个分区的 1-2 句话 + AI 自动出标签），对平台而言数据质量高

**边界**：
- 用户可以在 `/me/groups` 看到每个分区加入时留下的 `joinSymptoms` 快照 + 当时 AI 给的标签（不能编辑，只能退出 → 重新加入覆盖）
- 用户退出分区（`leftAt=now()`）后，快照保留；`User.symptomsText` / `User.aiDiseaseTags` **不回滚**（已回写的数据视为用户知识库沉淀）
- **为什么限 3**：引导用户聚焦主要疾病，减少信息过载和评论稀释；3 是用户拍板值
- **AI 分析失败**（DeepSeek 超时/错误）：降级为"允许用户跳过 AI 分析直接加入"（只存 `joinSymptoms`，`joinDiseaseTagsJson=null`）；不阻断主流程

### 5.7 社区 AI 审核（DeepSeek）

- 发帖/评论提交时：
  1. 服务端先跑现有敏感词词库（M5/M6 已有）→ 若 high 级命中直接拦截
  2. 否则调 DeepSeek `scenario=community_review`
     - prompt（system）：
       > 你是社区内容审核员。判断以下内容是否适合发布在患者病友互助社区。
       > 只输出 JSON：`{"result": "pass|reject|review_needed", "confidence": 0.95, "reason": "简短理由"}`
       > 判断标准：
       > - 医疗诊断/处方建议 → reject（理由：医疗建议不应来自非专业人员）
       > - 虚假广告/推销 → reject
       > - 明显辱骂/人身攻击 → reject
       > - 争议性话题或不确定 → review_needed
       > - 正常分享/提问/互助 → pass
  3. result=pass 且 confidence ≥ 0.8 → 自动发布（`reviewStatus=approved`）
  4. result=reject 且 confidence ≥ 0.8 → 自动拒绝（`reviewStatus=rejected`，告知用户）
  5. 其他情况（包含 review_needed 或低置信度） → `reviewStatus=pending`，进人工队列
- 所有 AI 判断字段写回 `CommunityPost.aiReviewResult/Confidence/Reason`
- **误杀率抽检 < 10%**：
  - 运营每周抽查 20 条 rejected 帖子，人工复核"真的该拒吗"
  - 抽查错杀率（误 reject 数 / 20）< 10% 达标
  - 若超标 → 调整 prompt 或降置信度阈值

### 5.8 审核队列 UI 改造

- `/admin/community/posts` 列表新增列：
  - **AI 判断**（pass/reject/review_needed 小徽章）
  - **AI 置信度**（0.92 / 0.75）
  - **操作**：
    - "认同 AI"：一键通过/拒绝，沿用 AI 结论
    - "覆盖"：选择相反决策 + 必填"覆盖理由"（保存到 `humanOverride`）
    - "查看 AI 理由"：抽屉展示 `aiReviewReason`
- 顶部过滤：
  - 置信度区间滑块（低置信度优先看）
  - 仅看 review_needed
  - 仅看被人工覆盖过的（用于后续调 prompt 参考）

### 5.9 AI 回复账号（"九泰 AI 助理"，用户拍板命名）

- 创建一个系统用户：phone=`__system_ai__`（特殊标记），`role=ai`, `isSystemAi=true`, displayName=**"九泰 AI 助理"**
- 触发场景（`AiAccountTemplate`）：
  - `new_user_welcome`：新用户注册成功且首次进入任一分区后，AI 在该分区下发一条欢迎评论（使用模板 + 昵称占位 + 分区名占位）
  - `faq_post_trigger`：用户发帖标题含"怎么报名"/"多久回复"/"能参加吗"等关键词时，AI 账号自动评论引用 FAQ（仅当该分区 `allowAiComment=true`）
- AI 账号的帖/评论**跳过 AI 审核**（内部信任），但打 `isAiGenerated=true` 标记，审核队列可过滤
- **分区级开关**：`CommunityGroup.allowAiComment=false` 时该分区内 AI 账号不触发任何自动评论（运营可针对敏感分区关闭）
- `/admin/ai-account` 管理页：
  - 模板开关
  - 触发规则编辑（JSON 可视化编辑，简化版）
  - 最近 50 次触发日志
- **合规红线**：AI 账号**绝不**给出医疗建议，模板内容只能是引导/FAQ

## 六、异常流

| 场景 | 行为 |
|---|---|
| `disease-matcher` / DeepSeek 超时（> 10s） | 前端提示"AI 分析超时，你可以直接跳过或重试"；**注册页**允许跳过（只存 symptomsText）；**加入分区弹窗**允许跳过 AI 标签直接加入（`joinDiseaseTagsJson=null`） |

| `DEEPSEEK_API_KEY` 未配置 | `disease-matcher.ts` 走 Mock 返回 3 条 fixture 标签（如"疑似心血管相关"），开发期照常可用 |
| 用户上传文件超 5MB | 前端 + 服务端双拦截，提示"单个文件不能超过 5MB" |
| 用户上传非法格式 | 拒收，提示"只支持 JPG / PNG / PDF" |
| 匹配助手 DeepSeek 超时 | 前端提示"AI 暂时不可用，你也可以直接浏览 [试验列表]"，不报错退出 |
| 匹配助手响应非 JSON | 记日志 + fallback 到"很抱歉，我没听清楚你的描述，可以再详细说说吗？" |
| 匹配助手 5 轮仍无结果 | 强制结束："根据你描述的内容，建议先咨询医生后再回来匹配试验" |
| 用户症状为空点匹配助手 | 允许进入，但开场白改为"先告诉我你的主要不适吧" |
| 用户加入第 4 个社区 | 拦截 + 提示"最多 3 个，请先退出一个" |
| 用户症状为空（`symptomsText`+`aiDiseaseTags` 都空）点加入分区 | 加入前弹窗强制要求填 `joinSymptoms`；提交后该分区的 AI 分析同步回写到 `User.aiDiseaseTags` |
| AI 审核 DeepSeek 超时 | 降级为 `reviewStatus=pending`，转人工 |
| AI 审核置信度 < 0.5 | 不信任 AI 结论，转人工 |
| 运营覆盖 AI 决策不填理由 | 表单必填拦截 |
| 医生分配失败（User 不存在） | 提示 + 日志 |
| AI 账号被用户 @ 或私信 | 本里程碑**不回复**（没实现即时对话），仅在发帖时触发模板 |
| 诊断文件被用户删除 | 软删除 30 天，期间运营能看到但标记"已删除" |

## 八、验收标准

### 8.1 用户能...

- 注册时能看到症状输入步骤，可跳过
- 输入症状后点"AI 分析"能在 5s 内看到 DeepSeek 返回的标签 chips（可删）
- 能上传确诊记录（图片/PDF）或跳过
- 能在 /match-assistant 对话 3-5 轮得到结构化试验推荐
- 能加入最多 3 个社区，加入前系统引导补症状
- 在社区发帖后能正常看到帖子（AI 通过时秒过）
- 能在 /me 看到自己的症状、上传文件、加入的社区
- 能看到帖子作者是医生（绿徽章）或 AI（灰徽章）

### 8.2 用户看到...

- 匹配助手页面顶部红底免责声明横幅 + 每条响应底部重复免责
- 社区首页看到"我加入的分区 / 推荐加入 / 其他可加入"三段布局（M8.3 路径反转，**保留**病种分区）
- 加入任何分区前都能看到"补充疾病信息"弹窗（含免责文案）
- 审核队列运营看到 AI 判断列 + 置信度

### 8.3 用户不能...

- 患者不能自称为医生（role 只能 admin 分配）
- 用户不能加入第 4 个社区
- 匹配助手不能说出"你可能得了 XX 病"（prompt 强约束 + 后端正则再过滤一遍，命中则兜底改写）
- 运营不能无理由覆盖 AI 决策
- 用户不能绕过敏感词 + AI 审核直接发帖（服务端强校验）

### 8.4 技术硬门槛

- `npx tsc --noEmit` 零错
- `npm run build` 通过，新路由全编译
- `disease-matcher.ts` 抽检准确率：用 5 个代表性症状描述跑（如"血压高 + 头晕"、"空腹血糖 8"、"乳腺术后化疗"），每次返回 ≥ 1 条合理标签（人工判断）；≥ 4/5 达标视为通过（决策 7 反转替代原"ICD-10 ≥ 20000 条"）
- 匹配助手 RT P95 < 5s（DeepSeek 云端往返）
- `disease-matcher` RT P95 < 5s
- 社区 AI 审核误杀率抽检 < 10%（运营每周抽 20 条）
- DeepSeek 未配置时 match-assistant / 社区审核 / disease-matcher 都降级到 Mock（开发期可用）
- 所有 LLM 调用必须写 `LlmCallLog`（sceneario 含 `disease_analysis`）
