## 二、用户故事（5 种身份）

### 游客（未登录）
- 作为一个对临床试验有兴趣但不想注册的访客，我想**先随意看 5 条试验详情**了解行情，看到第 6 条时网站明确告诉我"再看就得注册"并把注册入口顶在眼前。
- 作为游客，我不希望网站**没有任何提示就挡我的路**——前 5 条里应该在详情页底部看到"还可以看 X 条"的残余提示。

### 注册用户（patient role）
- 作为一个正在到处对比试验的患者，我可以正常浏览大量试验（目标：≤30 min 看 ≤20 条），**系统不该把我当成爬虫**封掉。
- 作为一个被误封的用户，我登录时能看到清楚的封锁页，**有明确的申诉按钮 + 输入框**让我说明情况（诸如"我是多发性硬化患者，确实对比了很多条"），提交后知道"预计 24 小时内答复"。
- 作为一个被封后申诉的用户，我**可以在 /me 或站内消息**看到运营的回复（通过/驳回 + 理由）。

### 医生（M8.3 引入，本里程碑仅预留 `role=doctor`）
- 医生目前不使用本里程碑的功能，但封号规则不区分角色（一并限流，但后台可白名单放行）。

### AI 账号（M8.3 引入）
- 本里程碑**不触发 AI 账号**相关逻辑。

### admin（后台运营）
- 作为运营，我要**看到所有被封锁的账号列表**（封锁时间、原因、是否申诉），快速筛"有申诉的"。
- 作为运营，我要能**点进一个封号**看完整行为日志（最后 30 min 看了哪 20 条试验），决定是误封还是真爬虫。
- 作为运营，我要能**一键解锁 + 回复申诉**，回复文案可以保存为模板。
- 作为运营，我要把**伦理批件号** `ethicsApproval` 填进试验信息后，**前台合规区能看到**（之前填了但没显示）。
- 作为运营，我要能**上传首页/社区/试验的配图**（图片素材库），上传后自动生成 URL 可引用；没素材时网站走默认占位不崩。

## 五、业务规则详解

### 5.1 游客 5 条限制

- 首次进入 `/trials/[slug]` 时，若无 `jt_guest_token` cookie，server action 生成 UUID 写 httpOnly cookie（`Max-Age=30d`、`SameSite=Lax`、`Secure` 生产开）
- 每次访问 `/trials/[slug]`（登录态且未被封 / 游客）写一条 `UserBehaviorLog`（`action=trial_detail_view`）
- **游客规则**：查询该 `guestToken` 在 `UserBehaviorLog` 里 `trial_detail_view` **去重 targetId** 的 count
  - `count <= 4`：正常渲染 + 底部加**残余提示**："你是游客，还可以免费查看 {5-count} 条试验详情"
  - `count == 5`：正常渲染 + 残余提示："这是最后 1 条免费详情，再看请先注册"
  - `count >= 6`：**不渲染详情**，返回"游客额度用完"页（沿用封锁页布局，CTA 按钮"去注册"→ `/auth`）
- **边界**：同一 trial 反复看不累加；只有看**不同** trial 才加 1

### 5.2 注册用户 30 min / 20 条封号

- 登录态访问 `/trials/[slug]`：
  1. 先查 `AccountLock` 该 userId 是否有 `unlockedAt = null` 的记录 → 有 → 重定向到 `/account-locked`
  2. 没有 → 写 `UserBehaviorLog`
  3. 查最近 30 min 内该 userId 的 `trial_detail_view` **去重 targetId** count
  4. `count >= 20` → 写 `AccountLock`（`reason="30min内查看20条试验详情，触发反爬规则"`，`triggeredBy="rate_limit:trial_detail_view"`），立即重定向到 `/account-locked`
- **封锁时长**：**无自动过期**，必须运营解锁（避免简单等待绕过）
- **白名单**：`OperatorUser` + `role=doctor` 的 `User` **不触发**封号（doctor 为 M8.3 引入的 role，此处提前预埋判断）

### 5.3 封锁页（/account-locked）

- 显示：
  - Headline：**"账号已被暂停使用"**（禁用"封号"这类生硬词）
  - 原因标签（从 `AccountLock.reason` 映射成友好文案）
  - 封锁时间（`lockedAt` 格式化）
  - 预计响应时效："运营会在 24 小时内回复你的申诉（工作日）"
  - 主 CTA："我要申诉" → `/appeal`
  - 辅助 CTA："拨打电话 400-xxx-xxxx"（复用现有客服电话占位）
- **禁止**：出现"爬虫""作弊""违规"等指责性用词（统一为"系统检测到异常频次"）

### 5.4 申诉表单（/appeal）

- 字段：
  - 手机号后 4 位（身份校验，对比 session.user.phone.slice(-4)，防止有人替别人申诉）
  - 申诉理由（必填，5-200 字）
  - "我已阅读并同意 [用户协议]" 勾选（必勾）
- 提交后：
  - 更新 `AccountLock.appealText` / `appealedAt` / `appealStatus="pending"`
  - 如果用户已经申诉过（`appealStatus !== "none"`）→ 提示"你已提交过申诉，等待运营处理中"
- 成功跳 `/appeal/submitted`，展示"已收到，工单号 `lock_xxxxx` 的前 8 位"

### 5.5 管理端 /admin/locks

- **列表**（`/admin/locks`）：
  - 列：用户（手机脱敏 138****1234 + 昵称） / 封锁时间 / 触发规则 / 申诉状态（none/pending/approved/rejected 的小徽章） / 操作（查看）
  - 顶部筛选：全部 / 有申诉 / 已解锁 / 处理中
  - 每页 20 条，按 `lockedAt desc`
- **详情**（`/admin/locks/[id]`）：
  - 用户基本资料（手机脱敏、昵称、注册时间、最近登录）
  - **行为日志**表格：近 30 min 查看的试验列表（trial title + slug + 时间 + 同一条去重标识）
  - 申诉内容（如果有）
  - 操作区：
    - **解锁按钮**：填写解锁理由（必填，保存到 `unlockReason`），点击后 `unlockedAt=now()`、`unlockedBy=operator.id`；如果之前有申诉，同步将 `appealStatus` 置为 `approved`
    - **驳回申诉按钮**：填写驳回理由 → `appealStatus=rejected` + `appealReplyText` + `appealHandledAt`（账号保持锁定）
  - 操作完成写 `AuditLog`（沿用 M6 审计底座）

### 5.6 伦理批件号前台展示

- `/trials/[slug]` 合规区（现有"责任单位""主办方"那块）末尾加一行：
  - 标签："伦理批件号"
  - 值：`ethicsApproval || '—'`（无时不隐藏，而是显示 "—"，让患者知道"这条没填"反而起保守作用；也可以运营视情况在后台不填时选择不显示——**待用户拍板**）
- 后台 `/admin/trials/[id]` 编辑表单已有 `ethicsApproval` 字段，无需改造

### 5.8 图片素材库 `/admin/media`（骨架）

- **本里程碑只做骨架**：列表 + 上传 + 默认占位策略
- 上传：
  - 接受 image/jpeg, image/png, image/webp, image/gif；单文件 ≤ 2MB
  - 上传后写 `public/media/{category}/{cuid}.{ext}`，记录入 `MediaAsset`
  - **不做**自动压缩（M8.2 或后续补）
- 列表：
  - 按 `category` 分页签（hero / community / trial / faq / avatar）
  - 每条显示：缩略图（64x64）+ 原文件名 + URL（一键复制）+ 尺寸 + 上传时间
  - 删除按钮（软删除，写 `isEnabled=false`——本模型暂无该字段，**待定：是否加字段或纯硬删**）
- **占位策略**（前台引用时）：
  - 首页 Hero、社区头图、试验详情配图：URL 为空/失效时 → CSS 渲染默认色块 + 图片描述文字（`alt` 内容居中显示）
  - **绝不**暴露"图片加载失败"（`onerror` 隐藏 img 并显示 fallback）

## 六、异常流

| 场景 | 行为 |
|---|---|
| 游客 cookie 被清 | 重新计数（允许重新获得 5 条配额；按设计默认接受该绕过可能，反爬核心覆盖登录用户，**已权衡**） |
| 登录用户 cookie 被清 | 不受影响（按 userId 计数） |
| 用户被封期间访问任意路由 | 全站中间件检测 → 除 `/account-locked` `/appeal` `/appeal/submitted` 外全部重定向到 `/account-locked` |
| 申诉理由 < 5 字 | 表单校验拦截 + 错误提示"说明一下遇到的情况（至少 5 字）" |
| 运营连续解锁多个账号（批量） | 本里程碑**不做批量**，一次一个（后续按运营反馈决定） |
| 伦理批件号字段为空 | 前台显示 "—"，不影响其他渲染 |
| 图片上传超 2MB | 服务端拒收，提示"图片大小不能超过 2MB，请压缩后再传" |
| 图片上传非法类型 | 服务端拒收，提示"只支持 JPG / PNG / WebP / GIF" |
| 图片链接失效 | 前台走占位色块 + alt 文字 |
| 行为日志表暴涨 | M8.1 阶段只保留 30 天（cron 删除 `createdAt < now() - 30d`），**cron 作业本里程碑不做**，留到 M6 审计日志同批处理（待定） |

## 八、验收标准

### 8.1 用户能...

- 游客能看前 5 条试验详情并看到"还可以看 X 条"的残余提示
- 游客第 6 条时能看到"去注册"页面而不是白屏或 500
- 注册用户正常浏览（< 20 条 / 30 min）不受影响
- 被封用户登录后能被重定向到封锁页，能一键进入申诉表单
- 被封用户能在 `/me` 看到自己的申诉处理结果（通过/驳回 + 理由）

### 8.2 用户看到...

- 封锁页没有"爬虫/作弊"等指责性用词
- 试验详情页合规区看得到"伦理批件号：{值}"或 "—"
- 图片不加载时显示色块 + 文字占位，**看不见**"图片加载失败"的破图

### 8.3 用户不能...

- 游客不能看超过 5 条试验详情
- 被封用户不能访问任何页面（除封锁页和申诉页）
- 用户不能重复申诉（二次打开申诉页提示"等待处理中"）
- 运营不能解锁时不填理由（必填校验）

### 8.4 技术硬门槛

- `npx tsc --noEmit` 零错误
- `npm run build` 通过，新增路由全编译
- `/account-locked` / `/appeal` / `/admin/locks` / `/admin/media` 路由 200
- 运营解锁操作有 `AuditLog` 记录
