## 0. 为什么做这个

M8.4 PRD 第九章拍板"接入位 4（社区分区头图）、接入位 5（试验详情顶图）拆到 M8.5"，理由是这两个接入位需要"每条记录挑一张图"——属于结构性变更，不能跟首页接入位混着做。M8.5 就是来还这两笔账的。

同时盘 M8.3 落地，发现两处"做了一半没接通"的状态：

1. **AI 欢迎评论**：`src/lib/ai-account.ts` 里的 `triggerWelcomeIfFirstVisit()` 函数已实现（fire-and-forget），AI 账号 + welcome 模板都已 seed，但**没人调它**——用户注册完成后，AI 不会自动出现，相当于没上线
2. **推荐分区**：`disease-matcher.ts` 的 `recommendGroups()` 已在 `/community` 首页用上，但用户**主动进社区才能看到**；新注册用户填完症状后，下一步直接跳到 /records 然后到 /me，**不知道有"推荐分区"这件事**——AI 帮匹配的能力没传达出去

第三件事是 M8.4 第九章把 `faq` 分类标记为"V1 不做独立 FAQ 页"，但 `/faq` 路由其实在 M4 就已落地，M8.4 的判断与现状不符。M8.5 顺手把 FAQ 配图位接通，让运营换 hero 不写代码。

合在一起就是 M8.5 的 **5 件事**——3 个素材库接入位、1 个 AI 触发挂载、1 个推荐分区弹窗。

**核心要求**：
- 不改 schema 大动作（最多每个目标表加一个 `coverMediaId` 可空字段）
- AI 触发严格 fire-and-forget，不能阻塞页面
- 弹窗对中老年用户友好，可关闭、可"稍后提醒"

## 二、详细规格

### 2.1 社区分区头图（接入位 1）

**前台展示**：
- 路由：`/community/[slug]`
- 位置：现有 `<section className="cm-hero">` **之上**新增一块横幅
- 比例：21:9（宽屏），高度移动端 160px / 桌面 280px，`object-fit: cover`
- 圆角：`var(--r-xl)`，与 `cm-hero` 视觉对齐
- **不在图上叠任何文字**——分区名 / 介绍仍在原 cm-hero 区域显示，避免文字与图叠加在中老年用户眼里看不清

**后台运营**：
- 在 `/admin/community-groups/[id]/edit`（或现有分区编辑入口）加一个字段"分区头图"
- 选择方式：弹一个素材选择器，显示 `MediaAsset.category='community' AND isEnabled=true` 的所有图，用缩略图网格展示，运营点一张即选中
- 提供"清除选择"按钮
- 不在分区编辑页内**直接上传新图**——让运营养成"先去 /admin/media 上传 → 再回来选"的两步流程，避免分区表单变重

**数据模型**：
- 在 `CommunityGroup` 表加 `coverMediaId String?`（外键弱关联：保留为字符串，运行时关联 MediaAsset）
- 不动现有 `headerImageUrl` 字段——它是 M8.3 加的字符串 URL 字段，本期视为废弃但不删（避免 migrate 风险），新逻辑只读 `coverMediaId`
- 写入 `headerImageUrl` 的老数据展示规则：`coverMediaId` 优先；为空时回落 `headerImageUrl`；都为空走降级

**降级**（无图时）：
- 不留空白 banner，直接不渲染顶部 banner 块——`cm-hero` 自然占位即可（与现状一致，保证向后兼容）

### 2.4 AI 欢迎评论触发挂载（接入位 4）

**触发点选定**：注册成功后第 3 步（/auth/register/records）的"完成并进入我的主页"按钮，**点击进 /me 路由的请求处理过程中触发**

**为什么不放在登录成功的 verifySmsCode action**：
- 登录时还没填症状，AI 欢迎评论里"建议你看看 X 分区"无内容可填
- 用户可能 30 天前注册过现在重新登录，触发欢迎会很怪

**为什么不放在 /me 首次访问**：
- /me 不知道用户是不是"刚走完注册"——会出现老用户每次进 /me 都被欢迎一次的 bug 风险

**实现位置**：
- 在 `/auth/register/records` 的 server action（处理"完成"按钮的那个 action）末尾加一段 `setImmediate(() => triggerWelcomeIfFirstVisit({...}))`——**fire-and-forget，不 await，不抛错阻塞重定向**
- 该 action 的 happy path：保存最后状态 → 触发欢迎（不阻塞）→ redirect("/me")
- 错误路径：欢迎触发失败不打断重定向，只在 server log 里 console.warn

**触发参数 `groupId` 怎么定**：
- 优先：用户填的症状 → recommendGroups → 第 1 名分区
- 兜底：取系统第 1 个 `isEnabled=true` 的分区（按 sortOrder asc）
- 都没有：跳过，不触发

**幂等性**：
- `triggerWelcomeIfFirstVisit` 函数本身已查询"是否已 commented"，重复调用安全
- 但还是要加一个用户级幂等：在 `User` 表加 `welcomeTriggeredAt DateTime?` 字段记录"已触发过一次欢迎"——避免用户反复跳转 /records → /me 多次触发
- 幂等检查在 server action 内做，不污染 ai-account.ts

**数据模型**：
- `User` 加 `welcomeTriggeredAt DateTime?`

**用户体感**：
- 用户进 /me 后大概 1-3 秒内（异步任务执行延迟），如果他这时去 /community/[他被推荐的分区]，会看到该分区最新一篇帖子下多了一条 AI 账号的欢迎评论
- /me 页本身**不显示任何"AI 欢迎你"的提示**——避免老人困惑"我不认识这个 AI 是谁"
- 用户进社区看到欢迎评论时，AI 账号头像旁标"AI 助手"标签（M8.3 已实现），用户能识别这是机器人而非真人

### 2.5 推荐分区弹窗（接入位 5）

**触发时机**：用户首次进 `/me` 时**自动弹一次**

**为什么是 /me 而不是注册完成那一步**：
- /auth/register/records 是"上传文件"步骤，弹推荐分区与上下文跳跃
- /me 是用户落地页，"在你的主页给你推荐"上下文自然
- 也避开了在注册流程里再加一步打断填写节奏

**为什么"首次"而不是"每次"**：
- 老人反复弹同一个弹窗会被劝退
- 弹一次后写 `User.recommendDismissedAt`，下次不弹
- 弹窗内提供"暂不需要"按钮——点了同样写时间戳，不再弹

**弹窗内容**：
- 标题：你可能想加入的病友圈
- 副标题（小字）：根据你刚才描述的症状，我们推荐了这几个分区，加入后就能看到病友讨论
- 主体：列出 1-3 个推荐分区卡片（来自 `recommendGroups` top 3）
  - 每张卡含：分区名、介绍前 30 字、"加入"按钮
  - "加入"按钮点击 → server action 写一条 `UserGroupMembership`，按钮变"已加入"，**不关闭弹窗**——让用户能连续加入多个
- 底部按钮：
  - 「都加入」（如有 ≥2 张推荐时显示）：批量加入所有推荐
  - 「先看看」（次要按钮）：关闭弹窗，写 `recommendDismissedAt = now()`
  - 关闭按钮 ×（右上角，44×44 大小）：等同「先看看」

**老人友好规则**：
- 弹窗内文字 ≥ 16px
- 按钮 ≥ 44×44，间距 ≥ 12px
- 弹窗背景遮罩点击关闭——但**默认关闭**这个行为（老人误触遮罩闭弹窗会迷茫）；只能从右上角 × 或底部按钮关闭
- ESC 键关闭——保留（键盘可达性）
- 弹窗打开时 6 秒内不允许任何按钮自动获得焦点之外的"诱导动作"（无自动跳转、无倒计时关闭）

**没有推荐时**：
- `recommendGroups` 返回空（用户跳过了症状填写、或 AI 没识别出标签）→ **不弹窗**，安静放过
- 弹窗只在"有 ≥1 个推荐"时才会出现

**强制 vs 可关**：
- **可关**——绝不强制；强制弹窗等于把用户当成羊群，违反"中老年友好"
- 但弹窗关闭后 /me 页右上角保留一个常驻入口"为你推荐 X 个分区"（小尺寸标签），点击重新打开弹窗——给后悔的用户一个回头路

**数据模型**：
- `User` 加 `recommendDismissedAt DateTime?`

**与 2.4 的关系**：
- 4 是"AI 在社区里给你打招呼"（被动看见）
- 5 是"在你主页弹个引导"（主动看见）
- 两者**不重叠**，可以并存——4 让用户进社区时不孤单，5 让用户知道"原来有社区可以进"

## 四、运营体验

### 4.1 运营要做的新事

1. 给社区分区配头图（**新流程**）
   - 第一步：进 /admin/media 上传一张 community 分类的图
   - 第二步：进 /admin/community-groups 编辑某个分区，弹素材选择器选这张图
   - 保存

2. 给试验配顶图（**新流程**）
   - 同上，分类选 trial，去试验编辑页选

3. 换 FAQ 配图（**最简单**）
   - 直接在 /admin/media 上传一张 faq 分类的图，前台自动顶替

4. 配置 AI 欢迎模板（**沿用 M8.3 入口**）
   - /admin/ai-templates 已存在，本期不改入口，只是触发时机变了
   - 运营可以通过编辑 `new_user_welcome` 模板的 `contentTemplate` 来改文案

### 4.2 运营不用做的事

- 推荐分区弹窗的内容**不需要运营手工配置**——直接复用 disease-matcher 的输出
- AI 欢迎触发**不需要运营手动开关**——通过模板 `isEnabled` 字段控制（M8.3 已有）

### 4.3 后台素材选择器组件（共用）

接入位 1、2 都需要一个"从素材库挑一张图"的组件，做成共用组件 `<MediaPicker category="community" value={...} onChange={...} />`：
- 弹窗呈现：3-4 列网格，每张图 120×120 缩略
- 顶部搜索框：按 `originalName` 模糊搜索
- 选中态高亮，底部"确认"按钮
- 已选图在表单上以缩略图 + 文件名展示，旁边"更换"和"清除"按钮

## 五、SEO / 可访问性

### 5.1 SEO

- 试验详情顶图：`<img alt={trial.title}>`，文案不重复"图片"二字
- 社区分区头图：`<img alt={`${group.name} 病友社区`}>`
- FAQ hero：`<img alt="常见问题" aria-hidden="true">`（装饰性，不计入 SEO）
- 试验顶图色块降级版：用 `<div role="img" aria-label={`${trial.disease} 病种`}>` 而非语义性 h2
- 不影响现有 OpenGraph、JSON-LD（保留）

### 5.2 a11y

- 推荐分区弹窗：
  - 容器 `role="dialog" aria-modal="true" aria-labelledby="rec-modal-title"`
  - 打开时焦点自动落在弹窗内第一个可交互元素（标题之后的"加入"按钮）
  - ESC 关闭
  - 关闭时焦点回到触发它的入口（首次自动弹时回到 /me 主标题）
  - 屏幕阅读器：弹窗打开时 `aria-live="polite"` 朗读"为你推荐 N 个病友分区"

- AI 欢迎评论：
  - AI 账号头像 `<img alt="AI 助手">`
  - "AI 助手"标签 `aria-label="AI 自动生成的评论"`

## 六、验收清单（产品视角）

**用户能看到 / 能做 / 不能怎样**：

1. ✓ 运营在 /admin/media 上传一张 community 图后，到 /admin/community-groups 选定该图给"乳腺癌"分区，刷新 /community/breast-cancer **能看到顶部头图**
2. ✓ 给"乳腺癌"分区清除头图后，刷新页面**回到原 cm-hero 占位**，不留空白 banner
3. ✓ 运营在 /admin/trials 给某条试验选定一张 trial 图，刷新 /trials/[slug] **能看到顶图**
4. ✓ 试验**没配图**时，trials/[slug] 顶部展示**病种色块** + 大字病种名，不留空白
5. ✓ 同一病种的不同试验，色块颜色**一致**（哈希稳定）
6. ✓ 运营在 /admin/media 上传一张 faq 图，刷新 /faq **看到 hero banner**；上传第二张 faq 图，自动顶替旧图
7. ✓ 新用户走完注册第 3 步进 /me 后，**3 秒内**到 /community/[他的推荐分区] 能看到该分区最新帖子下有一条 AI 账号的欢迎评论
8. ✓ 同一用户反复点 /records 完成按钮多次进 /me，**只触发一次**欢迎评论
9. ✓ 用户没填症状（跳过第 2 步），进 /me 后**也能正常打开**，不报错；不弹推荐分区弹窗
10. ✓ 用户填了症状且 disease-matcher 有结果，第一次进 /me **自动弹推荐分区弹窗**
11. ✓ 弹窗中点"加入"按钮，按钮变"已加入"，弹窗**不关闭**，可继续加入下一个
12. ✓ 弹窗中点"先看看"或右上角 ×，弹窗关闭，刷新 /me **不再自动弹**
13. ✓ 弹窗关闭后 /me 顶部有"为你推荐 N 个分区"小入口，点击**重新打开**弹窗
14. ✓ 推荐结果为空时，**不弹窗，无入口**
15. ✓ AI 欢迎触发失败（DB 写入超时等）**不阻塞** /me 重定向，用户体感正常
16. ✓ `pnpm tsc --noEmit` 通过；`pnpm build` 通过
17. ✓ 试验详情页 LCP 不劣化超过 150ms（顶图 lazy load 不应影响首屏）

## 八、技术实现要点（高层）

### 8.1 数据模型变更

```prisma
model CommunityGroup {
  // ...
  coverMediaId String?  // 新增：关联 MediaAsset.id
  // headerImageUrl 保留不动（向后兼容，新逻辑不读）
}

model ClinicalTrial {
  // ...
  coverMediaId String?  // 新增
}

model User {
  // ...
  welcomeTriggeredAt    DateTime?  // 新增：AI 欢迎已触发时间戳，幂等保证
  recommendDismissedAt  DateTime?  // 新增：推荐弹窗已关闭时间戳，不再自动弹
}
```

**Migration 名**：`m85_cover_media_and_user_flags`

**注意**：
- `coverMediaId` 不用 prisma `@relation`——保持弱关联，避免删除 MediaAsset 时级联爆炸
- 运行时由 lib 函数 `getMediaById(id)` 主动查
- 字段全为可空，老数据自然为 null，零迁移成本

### 8.2 共用 lib 函数（约定）

- `getMediaById(id: string | null): Promise<MediaAsset | null>`：弱关联读取
- `getLatestFaqHero(): Promise<MediaAsset | null>`：FAQ 单图最新一张
- `getDiseaseColorClass(disease: string): string`：试验色块降级，返回 6 选 1 的 CSS 类名

具体放 `app/src/lib/media.ts`（M8.4 已建）扩展。

### 8.3 fire-and-forget 触发模式

- /auth/register/records 的"完成"server action 末尾用 Node 的 `queueMicrotask` 或 `setImmediate` 调度 `triggerWelcomeIfFirstVisit`
- **不**用 `Promise.race` / `Promise.allSettled`——这两者会让重定向等待异步任务，违反 fire-and-forget
- **不**直接 `triggerWelcomeIfFirstVisit(...)` 不 await——Next.js 16 server action 在 response 发出后任务可能被进程回收，需要显式调度
- Tech-lead 拿到这条要去查 `app/node_modules/next/dist/docs/` 里 server action 的执行模型（这是 Next 16 不是老版本）

### 8.4 弹窗组件

- Client Component，懒加载（`dynamic(() => import('./RecommendModal'), { ssr: false })`）
- 数据由 /me 的 server component 预先算好（`recommendGroups` 的结果）作为 props 传入
- 弹窗显示判断：`!recommendDismissedAt && recommended.length > 0`
- 加入分区的 server action 复用 M8.3 已有的 `joinGroupAction`，本期不重写

### 8.5 性能护栏

- 试验详情顶图、社区分区头图必须 `loading="lazy"`（不在首屏关键路径）
- FAQ hero 在首屏关键路径，`loading="eager"` + `fetchpriority="high"`
- 弹窗组件 lazy import，不影响 /me 首屏
- 病种色块用 CSS 类实现，不引 JS
