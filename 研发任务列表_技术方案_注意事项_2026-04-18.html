⭐ 第一阶段：主链路闭环（13项，3-4周）

这个阶段的目标是跑通"患者找试验 → 提交预筛 → 后台跟进"的完整链路。

1.1 患者端基础框架与首页 第一阶段
  • 创建 React/Next.js 项目框架
    包含：路由结构、全局样式、导航组件、响应式布局
  • 设计全局导航栏
    包含：首页、找试验、患者社区、患者帮助、联系我们、登录入口
  • 实现首页（Hero + 病种入口 + 推荐试验 + 流程说明）
    验收：首页可展示 4-5 个重点病种，3-5 个推荐试验，平台价值说明清晰可见
1.2 试验发现能力 第一阶段
  • 实现试验列表页（列表 + 筛选）
    筛选维度：病种/适应症、城市、招募状态；显示结果数量
  • 实现试验详情页
    包含：标题、摘要、患者友好说明、适合人群、不适合人群、参与流程、位置、预筛按钮
  • 创建病种/适应症标签体系
    兼容传统疾病（肺癌、糖尿病等）+ 医美适应症（脱发、黄褐斑等）
1.3 患者登录与账户 第一阶段
  • 实现登录/注册页
    方案：简化版手机号登录（可用伪验证码快速实现，正式上线再对接短信服务）
  • 实现患者基本资料页
    字段：姓名、性别、年龄、手机号、所在城市、主要病种；保存后可用于表单预填
  • 实现登录回跳机制
    登录后返回用户原始目标页（预筛表单、社区发帖等）
1.4 预筛表单与提交 第一阶段
  • 设计预筛表单数据结构与字段类型
    类型：单行文本、多行文本、单选、多选、是/否、数字、日期
  • 实现项目级预筛表单页（前端+后端）
    能力：加载特定项目表单 → 自动预填基础资料 → 填写项目问题 → 校验 → 提交
  • 实现预筛提交成功页
    展示：感谢信息、后续跟进说明、FAQ 入口、社区入口
1.5 后台基础框架与试验管理 第一阶段
  • 创建后台登录页和权限体系
    角色：operator（运营）、admin（管理员）；简化版权限，无细粒度 RBAC
  • 实现试验列表与编辑页
    能力：新建、编辑、筛选、复制已有试验；字段包括标题、说明、城市、病种、状态等
  • 实现预筛表单配置页
    能力：从固定字段库选择 → 配置顺序、必填、说明 → 预览 → 发布
1.6 线索池与跟进 第一阶段
  • 实现线索生成逻辑
    预筛提交时自动创建线索记录，包含患者资料、答案、来源、时间
  • 实现线索列表与筛选
    筛选：试验、病种、城市、提交时间、状态；显示患者名（脱敏）、手机号、试验名等
  • 实现线索详情与跟进记录
    能力：查看患者资料 + 预筛答案 + 来源 + 状态流转 + 录入跟进时间/方式/备注

⚙️ 技术实现方案

推荐技术栈

前端框架

Next.js（带 TypeScript）

  • App Router for modern routing
  • Server Components for performance
  • Built-in API routes for backend integration
  • SSR/SSG for SEO-friendly content

UI 组件库

Shadcn/ui + Tailwind CSS

  • Production-ready component set
  • Fully customizable
  • Responsive design out of box
  • Dark mode support

后端框架

Node.js + Express 或 NestJS

  • RESTful API design
  • Middleware for auth & logging
  • Easy to deploy & scale
  • Rich ecosystem

数据库

PostgreSQL（推荐）或 MySQL

  • Relational data integrity
  • Strong ACID compliance
  • Excellent for structured data
  • Built-in JSON support

ORM/Query Tool

Prisma 或 TypeORM

  • Type-safe database access
  • Auto migrations
  • Schema introspection
  • Great DX

认证方案

NextAuth.js（简化版）或自建

  • Session-based auth for V1
  • Easy integration with Next.js
  • Support multiple providers
  • Simple but secure

部署方案

Vercel（前端）+ 自建服务器或云（后端）

  • Vercel: Next.js native, auto scaling
  • Backend: Docker + Docker Compose
  • Database: Managed PostgreSQL service
  • Single command deployment

事件埋点

PostHog 或 Mixpanel

  • Easy frontend SDK integration
  • Built-in funnel analysis
  • Session replay capability
  • Privacy-first options available

API 设计原则

RESTful 约定

患者端 API：

  • GET /api/trials - 获取试验列表
  • GET /api/trials/:id - 获取试验详情
  • POST /api/trials/:id/prescreen - 提交预筛
  • GET /api/me/profile - 获取个人资料
  • GET /api/me/applications - 获取报名记录

后台 API：

  • GET /api/admin/trials - 试验列表
  • POST /api/admin/trials - 创建试验
  • GET /api/admin/leads - 线索列表
  • PATCH /api/admin/leads/:id - 更新线索状态

所有请求返回统一格式：

{ "code": 0, "message": "success", "data": {...} }

安全与合规要点

🔐 必须实现
  • 会话管理：用户登录后必须通过 JWT 或 Session Cookie 验证，后端每次 API 调用验证身份
  • 手机号脱敏：后台列表页展示手机号时必须脱敏（如 138****8888）
  • 敏感词过滤：社区内容发布前必须过滤敏感词，记录风险提示供审核人员参考
  • 审计日志：所有关键操作（试验创建/编辑、线索状态变更、审核操作）必须记录操作人、时间、变更内容
  • HTTPS：生产环境必须使用 HTTPS，禁止明文传输任何敏感数据
  • 表单校验：前端和后端都必须校验用户输入（不能只在前端校验）

性能优化策略

⚡ 快速启动的优化
  • 数据库索引：在 trial_id、patient_id、created_at 等频繁查询字段上建索引
  • API 缓存：试验列表、FAQ、社区分区等不常变的数据用 Redis 缓存，5-10 分钟过期
  • 前端优化：Next.js Image 组件优化图片、代码分割减少 JS 包体积
  • 分页加载：列表页默认每页 20-50 条，避免一次性加载大量数据
  • 数据库分离：读写分离不是首版重点，但可预留扩展空间

⚠️ 项目搭建过程中的关键注意事项

🚨 高优先级注意事项（必须做）

1. 数据流闭环要验证清楚

问题：患者提交预筛后，后台线索没有生成或显示错误。

解决方案：

  • 在第一阶段 Week 2 就要完成"患者端提交 → 后台线索生成"的端到端测试
  • 不能等到第一阶段结束才发现这个问题
  • 工程师自测时必须包括：登录患者 → 提交预筛 → 后台看到线索这个完整流程
2. 预筛表单的版本管理要做对

问题：运营修改了项目 A 的预筛表单，但患者之前的答案展示错乱。

解决方案：

  • 每次发布新版预筛表单时，必须生成版本号（v1, v2, v3...）
  • 患者提交时记录当前表单版本号
  • 后台查看答案时，始终按照提交时使用的版本号来解析答案
  • 不能因为表单变更就改变历史答案的解读方式
3. 试验状态与前台可见性的映射要明确

问题：运营下线了某个试验，但患者端仍然能看到"立即预筛"按钮。

解决方案：

  • 定义清晰的状态机：试验必须同时满足 publish_status=已发布 AND recruitment_status=招募中，才能在前台显示预筛入口
  • 任何一个条件不满足，预筛按钮都应该被隐藏或变为"暂时无法预筛"
  • 后端 API 返回试验详情时，必须包含 can_prescreen=true/false 标志
4. 病种/适应症的多表关系要处理好

问题：一个试验可能关联多个病种标签（如"肺癌"和"脑转移瘤"），但数据库查询时容易出现重复或遗漏。

解决方案：

  • 用关联表 trial_tag 连接 trial 和 disease_tag，支持多对多
  • 前端分页列表和后台列表在展示时，注意 DISTINCT 或分组，避免同一个试验显示多次
  • 社区分组也要对应到 disease_tag，每个分区一个病种，确保清晰
5. 登录回跳要实现正确

问题：患者点击"预筛"按钮 → 进入登录页，登录后应该回到预筛表单页，但回到了首页。

解决方案：

  • 登录页接收 redirect_to 参数（或从 referer 推断）
  • 登录成功后，如果 redirect_to 存在，跳转到该地址；否则跳转到首页
  • 常见的回跳目标：/trials/:id/prescreen, /community/new, /me/profile

⚡ 中等优先级注意事项（强烈建议做）

6. 手机号登录的实现方案

当前状态：PRD 中提到"产品层按手机号设计，技术层可先采用简化认证方案"。

建议方案：

  • V1 实现（快速启动）：用伪验证码（模拟短信，后端直接接受任何 6 位数字）
  • 患者输入手机号 → 前端发送 /api/auth/send-code → 后端返回伪验证码给前端（仅开发环境） → 患者输入验证码 → 生成 Session/Token
  • 生产前（第一阶段结束前）对接真实短信服务（阿里云/腾讯云等）
  • 不要在 UI 上隐藏伪验证码逻辑，这会造成测试困难；改为让测试人员知道验证码规则即可
7. 患者资料预填的时机

关键场景：已登录患者进入预筛表单，基础资料字段应该自动填充。

实现细节：

  • 后端 GET /api/trials/:id/prescreen-form 时，返回当前用户的 patient_profile 数据
  • 前端渲染表单时，如果字段 is_profile_field=true 且当前用户已登录，则将 profile 数据填入该字段
  • 但要让用户可以修改预填的数据（不能只读）
  • 患者修改资料后提交预筛，该提交答案应该自动更新患者 profile（可选优化）
8. 社区与试验详情的 CTA 位置要突出

问题：用户在社区看到帖子，想报名相关试验，但找不到"立即预筛"按钮。

解决方案：

  • 社区帖子详情页底部或侧边栏必须显示相关试验卡片 + 立即预筛按钮
  • 社区分区页也要展示本分区关联的试验列表 + 预筛入口
  • 这不仅是 UX 问题，也是转化漏斗的关键
9. 审核状态的友好映射

后台数据库状态：post_review_status 有 4 种：待审核、已通过、已驳回、已隐藏

患者端展示规则：

  • 已通过 → 公开展示
  • 待审核 → 显示"审核中...请耐心等待"
  • 已驳回 → 显示"您的帖子未符合社区规范，已隐藏"（不要展示"驳回"这个冷冰冰的词）
  • 已隐藏 → 隐藏（不显示或显示"已隐藏"）

📋 低优先级注意事项（未来优化）

10. 线索自动分配（先不做）

当前实现：运营人工在后台指定线索负责人。

V2 方向：可以加入自动分配规则（按 operator 的当前负责线索数均衡分配等），但 V1 不做。

11. 复杂条件跳转（先不做）

当前实现：预筛表单支持 7 种字段类型，但不支持"如果选择了 A，则显示 B"这样的条件逻辑。

理由：会大幅提升复杂度，不适合短周期 Vibe Coding。V2 再考虑。

12. 多租户和研究中心后台（先不做）

当前实现：单一平台、单一运营后台。研究中心的协作信息通过运营人员在线索详情页人工录入。

理由：建设研究中心后台会引入权限、数据隔离等复杂问题。V1 不需要。

⏰ 常见时间陷阱

⏱️ 容易低估的工作量
  • 响应式布局适配：UI 在 PC 和手机上看起来一样简单，但适配移动端的边界情况很多，通常需要 15-20% 的额外工时
  • 表单校验和错误提示：从"能提交"到"体验友好的表单"，差异很大，包括前端校验、后端校验、错误信息的清晰度
  • 列表筛选排序分页：简单的分页容易实现，但加上多个筛选条件后，SQL 查询优化和性能成为瓶颈
  • 数据脱敏和安全检查：不只是隐藏手机号，还要考虑 XSS、SQL 注入、CSRF 等防御，加法规检查代码往往需要 3-5 天
  • 审计日志实现：看起来简单（记录谁在什么时间做了什么），但要确保不遗漏关键操作、性能不下降，需要仔细设计

  • 所有关键操作有审计日志记录

❓ 常见问题

Q1：能否先做传统疾病项目，后续再支持医美适应症？

答案

不建议。建议从一开始就设计病种标签体系兼容两类。理由是：

  • 数据库 schema 和前端筛选逻辑都一样，没有额外复杂度
  • 早期支持能为未来客户拓展做准备
  • 如果后续再改，会涉及迁移历史数据，成本更高

Q2：第一阶段 3-4 周的估算是否现实？

答案

假设 1 名有经验的全栈工程师，这个估算是现实的，但需要：

  • 需求（PRD）在开发前最终确认，不要在开发中频繁改需求
  • 工程师可以全职投入，不被其他项目打断
  • 产品或设计在关键集成节点（Week 2、Week 3）及时验收反馈
  • 测试环境和数据库环境可以快速搭建

如果是初级工程师，建议预留 +30% 时间（4-5 周）。

Q3：为什么推荐 Next.js 而不是其他框架？

答案

对于这个项目的最优选择：

  • Next.js 优势：前后端可以用一套框架，API routes 支持简单后端，SSR 对 SEO 友好（试验详情页需要）
  • 如果全分离：React + Express，学习成本和部署复杂度都会增加
  • 如果用全栈框架：Remix、SvelteKit 也不错，但 Next.js 生态更成熟

简单说：用 Next.js 可以一个工程师快速搞定前后端，减少协调成本。

Q4：社区内容如何审核？需要机器学习吗？

答案

V1 不需要 ML，用规则即可：

  • 本地敏感词库：维护一份敏感词列表（联系方式、违规医疗词汇等），用正则或字符串匹配检测
  • 自动拦截：内容发布时如果命中敏感词，标记为"待审核"而不是直接拒绝
  • 人工审核：运营人员每天登录后台审核工作台，通过/驳回/隐藏，处理速度快（5-10 分钟一条）
  • V2 方向：如果审核量爆炸，再考虑接入第三方内容审核 API（阿里云、腾讯云等）

Q5：为什么不做 App 版本？

答案

短周期和 ROI 考虑：

  • V1 用 Web 响应式方案，工作量 100%；加 iOS/Android 各需要额外 50%+ 工时
  • 临床试验患者通常用浏览器访问（搜索、链接分享），不是高频 App 用户
  • Web 版本用 iOS PWA 或微信小程序也能接近 App 体验，投入更低
  • 如果 6 个月后数据表明需要 App，再专门开发，到时候 Web 代码已经稳定

Q6：如何处理"一个患者重复提交同一项目"？

答案

设计规则：

  • 当前实现：允许重复提交，每次提交生成新线索，后台可查看该患者的所有历史提交
  • 防止误操作：前台"我的报名"页展示已提交的项目，患者可以看到自己之前报过哪些
  • 不强制去重：因为可能是不同时间点、不同健康状态的重新评估，都应该保留
  • 后台视角：运营可以选择合并线索或创建新的，不强制系统自动处理

Q7：预筛表单字段能否扩展？例如增加文件上传？

答案

V1 不做文件上传，理由：

  • 增加复杂度：需要文件存储、病毒扫描、存储容量管理
  • 医疗合规性：文件涉及患者隐私数据，需要加密存储和安全审计
  • 实际需求不强：早期预筛只需要基本信息，真正需要病历的场景交给线下协同
  • V2 方向：如果运营反馈患者无法在线提供必需信息，再加文件上传能力

Q8：后台账号能否支持微信/钉钉登录？

答案

不建议 V1 做，原因：

  • 增加依赖（微信、钉钉的 OAuth）和配置复杂度
  • 运营人员通常没有企业微信账号，反而降低可用性
  • 用传统用户名/密码，运营可以快速申请账号，更灵活
  • V2 方向：如果企业已经全面使用钉钉或企业微信，再做单点登录

Q9：线索导出为 Excel 是否是 V1 需求？

答案

根据 PRD 的 Open Questions，这点尚未决策。建议：

  • 如果运营说"必须"：在 Week 2 的集成点加入简单导出功能（勾选线索 → 生成 CSV）
  • 如果没有硬需求：放到 V1.1（上线 2 周后基于反馈），先保证线索管理的核心功能可用
  • 导出本身不难（后端查询 + CSV 生成），但要考虑数据量大时的性能和权限

Q10：如何保证数据安全，特别是患者隐私？

答案

分层保护：

  • 传输层：生产环境必须 HTTPS，禁止明文 HTTP
  • 存储层：患者手机号等敏感字段可考虑加密存储（如 AES），但不是首版必须
  • 展示层：后台列表脱敏手机号（138****8888），不要显示完整号码
  • 访问控制：后台操作需要登录认证，线索数据只有有权限的运营可见
  • 审计：所有关键操作记录日志，包括谁在什么时间访问了哪些患者数据
  • 合规：在用户协议和隐私政策中声明数据使用方式，用户同意才能使用
