## 文档摘要

本 PRD 用于定义“临床试验患者招募网站”V1 的统一运营后台。

它的目标不是建设完整 CTMS、CRO 项目管理系统或研究中心门户，而是建设一套内部可用、短周期可落地的统一后台，帮助团队在同一个工作台中完成以下核心任务：

1. 维护公开站的试验项目信息。
2. 配置项目级预筛表单。
3. 统一承接和跟进患者线索。
4. 维护 FAQ 与患者教育内容。
5. 审核患者社区内容，并把社区流量导向预筛。
6. 查看最小可用的经营与转化数据。

本版明确遵循以下实现边界：

1. 后台只保留 `operator` 与 `admin` 两类角色。
2. 社区是 V1 强需求模块，但在后台里仅作为一个内容与审核模块存在，并非后台主体。
3. 研究中心协同在 V1 中只做人工记录，不建设独立研究中心后台。
4. 项目级预筛表单采用“受控配置”模式，不做通用低代码表单平台。
5. 整体范围必须适合短周期 `vibe coding` 落地。

---

## 一句话定位

一个面向内部运营团队的统一后台，用来支撑“试验上线 - 预筛配置 - 线索处理 - 内容审核 - 基础统计”这条招募闭环。

---

## Problem Statement

如果没有统一运营后台，临床试验患者招募网站在 V1 阶段会遇到以下问题：

1. 试验信息、FAQ、社区内容、预筛配置和患者线索散落在不同文档或工具中，运营无法形成统一闭环。
2. 新试验上线需要研发频繁介入，导致项目发布速度慢，不适合短周期持续上新。
3. 患者提交预筛后，如果没有统一线索池和状态流转，运营很难判断哪些线索需要优先联系，哪些已经关闭。
4. 患者社区内容如果没有审核工作台，会带来敏感信息外露、违规宣传、低质量内容堆积等风险。
5. 没有最小可用看板，团队无法快速知道哪些试验、哪些来源、哪些社区内容真正推动了预筛提交。
6. 如果后台一开始就做成复杂多角色、多工作流、多门户系统，会明显超出 V1 的实现能力，不利于 `vibe coding` 交付。

因此，V1 需要一套范围清晰、模块收敛、数据闭环明确的统一运营后台。

---

## North Star Connection

平台级 North Star：

`Weekly Qualified Prescreen Submissions`

运营后台对 North Star 的贡献路径如下：

1. 试验管理让更多项目能更快上线，增加可转化供给。
2. 项目级预筛配置让患者看到更贴近项目要求的问题，提高有效提交率。
3. 线索池和跟进记录让团队更快处理新线索，提高有效转化效率。
4. FAQ 与社区审核提升患者信任，减少疑虑，提升预筛启动率与提交率。
5. 基础统计让团队知道哪个项目、哪个内容、哪个入口更有效，从而持续优化。

模块级核心假设：

- 若运营可以在一个后台内完成试验、表单、线索和内容管理，则试验上线效率与线索响应效率都会显著高于“人工分散处理”方式。
- 若社区和 FAQ 的审核、推荐、导流都纳入同一后台，则“内容信任层 -> 预筛转化层”会更稳定。

## Target Users

### 主要用户

1. 运营用户 `operator`
   目标：维护试验、配置项目级预筛表单、处理线索、审核社区内容、维护 FAQ、查看关键数据。

### 次要用户

1. 管理员 `admin`
   目标：管理后台账号、维护基础配置、查看全局数据、处理少量系统级设置。

### 角色收敛原则

1. `operator` 承接招募运营、内容运营、版主、审核员的大部分日常工作。
2. `admin` 是平台内部少量高级账号，拥有 `operator` 的全部能力，并额外负责账号与系统配置。
3. V1 不拆更细的权限矩阵，不设计模块级、字段级、按钮级复杂权限。

---

## Scope

### In Scope

1. 后台登录。
2. 工作台首页。
3. 试验管理。
4. 项目级预筛表单配置入口与发布。
5. 线索池与线索详情。
6. 线索状态流转与跟进记录。
7. 人工记录研究中心协同信息。
8. FAQ 管理。
9. 社区分区配置。
10. 帖子与评论审核。
11. 基础统计看板。
12. 基础系统配置。
13. 关键操作审计日志。

### Out of Scope

1. 研究中心独立登录后台。
   原因：V1 先由平台运营人工承接中心协同。
2. 自动分配中心、自动分配线索、自动消息触达工作流。
   原因：流程引擎复杂度高，短周期内不必要。
3. 完整 CTMS、合同管理、预算管理、访视管理。
   原因：不属于患者招募网站 V1 核心任务。
4. 复杂 RBAC 权限体系。
   原因：`operator/admin` 已能支撑 V1。
5. 自定义 BI 报表平台。
   原因：V1 只需固定统计指标。
6. 社区私信、官方群聊、直播运营台。
   原因：社区在 V1 中只是辅助模块。
7. 通用低代码表单搭建器。
   原因：项目级预筛表单已另行定义为受控配置能力。

## User Stories

1. 作为 `operator`，我希望能快速新建和发布试验，这样新项目可以尽快上线开始招募。
2. 作为 `operator`，我希望能在试验详情里直接配置项目级预筛表单，这样我不用反复找研发改页面。
3. 作为 `operator`，我希望所有预筛提交都进入统一线索池，这样我能按优先级处理新线索。
4. 作为 `operator`，我希望在线索详情里看到患者基础资料、项目答案和来源页面，这样我能判断下一步跟进动作。
5. 作为 `operator`，我希望记录每次电话、微信或人工沟通情况，这样团队成员能看到完整上下文。
6. 作为 `operator`，我希望在后台里处理 FAQ 和社区审核，这样患者看到的内容可信、可控、可导流。
7. 作为 `operator`，我希望知道社区内容有没有带来预筛启动，这样我能判断社区运营是否有效。
8. 作为 `admin`，我希望统一管理后台账号和基础配置，这样系统不会因为配置散乱而失控。

## Functional Requirements

### 1. 登录与权限

1. 系统必须支持 `operator` 与 `admin` 通过统一登录页进入后台。
2. V1 必须只保留 `operator` 与 `admin` 两类后台角色。
3. `operator` 必须可访问工作台、试验管理、预筛表单配置、线索池、FAQ 管理、社区配置、社区审核、基础统计。
4. `admin` 必须拥有 `operator` 全部能力，并额外可访问后台账号管理、敏感词配置、基础标签配置等系统配置。
5. V1 不要求复杂权限编排；除 `admin` 专属配置能力外，其余业务模块默认对 `operator` 开放。
6. 所有后台关键操作必须记录操作者、时间和操作对象。

### 2. 工作台首页

7. 工作台首页必须展示今日新增线索数。
8. 工作台首页必须展示待联系线索数。
9. 工作台首页必须展示待审核帖子 / 评论数。
10. 工作台首页必须展示当前招募中的试验数。
11. 工作台首页必须展示最近 7 日预筛启动数与提交数。
12. 工作台首页必须提供快速入口，至少包括：
   - 新建试验
   - 查看新线索
   - 查看待审核内容
   - 新建 FAQ
13. V1 工作台首页不要求支持自由拖拽、自定义组件布局。

### 3. 试验管理

14. 系统必须支持试验列表页，展示所有试验项目。
15. 试验列表必须支持按试验名称、病种 / 适应症、城市、招募状态筛选。
16. 试验列表默认按最近更新时间倒序展示。
17. 试验列表行内至少应展示：
   - 试验名称
   - 病种 / 适应症
   - 招募城市
   - 招募状态
   - 公开状态
   - 当前表单状态
   - 最近更新时间
18. 系统必须支持新建试验和编辑试验。
19. 试验编辑页至少必须支持以下字段：
   - 试验标题
   - 简短摘要
   - 患者友好型试验说明
   - 适合人群
   - 不适合人群
   - 参与流程说明
   - 城市 / 研究地点
   - 招募状态
   - 公开状态
   - 病种 / 适应症标签
20. 试验必须支持以下基础状态：
   - 草稿
   - 招募中
   - 暂停招募
   - 已关闭
21. 公开状态必须至少支持：
   - 未发布
   - 已发布
   - 已下线
22. 试验必须支持关联 FAQ 内容。
23. 试验必须支持关联病种 / 适应症社区。
24. 试验必须支持配置是否在首页或社区中作为推荐项目展示。
25. V1 应支持复制已有试验作为新试验的起点，以提高上新效率。
26. V1 不要求支持多中心复杂分组、招募名额管理或排期甘特图。

### 4. 项目级预筛表单配置

27. 每个试验必须在后台存在“预筛表单配置”入口。
28. 预筛表单配置能力必须复用 [项目级预筛表单PRD_V1_2026-04-18.md](</e:/Projects/03_患者招募/项目级预筛表单PRD_V1_2026-04-18.md>) 中定义的字段模型与发布规则。
29. 后台必须允许运营从固定字段组件库中为试验配置基础资料字段与项目特有字段。
30. 后台必须支持预筛表单的预览。
31. 后台必须支持预筛表单发布。
32. 系统必须保证任一试验同一时刻仅存在一个“当前生效”的预筛表单版本。
33. 试验列表与试验详情中必须显示当前表单状态，至少包括：
   - 未配置
   - 草稿中
   - 已发布
34. 若试验已设为“招募中”，但尚未发布预筛表单，后台必须显著提示风险。
35. V1 不支持通用低代码表单搭建器、复杂条件逻辑和多版本并行投放。

### 5. 线索池

36. 预筛提交后，系统必须在后台生成统一线索记录。
37. 线索池必须支持列表查看所有线索。
38. 线索池必须支持按以下条件筛选：
   - 试验
   - 病种 / 适应症
   - 城市
   - 来源页面
   - 线索状态
   - 负责人
   - 提交时间范围
39. 线索池默认必须按提交时间倒序排列。
40. 线索列表至少应展示：
   - 患者姓名或匿名标识
   - 手机号脱敏展示
   - 试验名称
   - 病种 / 适应症
   - 来源页面
   - 提交时间
   - 当前状态
   - 负责人
41. 系统必须支持在线索列表中快速修改线索状态。
42. 系统应支持为线索手动指定负责人。
43. V1 不要求自动分配负责人。
44. V1 不要求复杂批量操作；如需导出，可在后续版本补充。

### 6. 线索详情与跟进记录

45. 线索详情页必须展示患者基础资料。
46. 线索详情页必须展示项目级预筛表单答案。
47. 线索详情页必须展示来源信息，至少包括：
   - 来源页面类型
   - 来源页面 ID
   - 若来自社区，则展示关联帖子 ID
48. 线索详情页必须支持以下线索状态流转：
   - 新线索
   - 待联系
   - 已联系
   - 跟进中
   - 关闭
49. 线索详情页必须支持录入跟进记录。
50. 跟进记录至少应包含：
   - 跟进时间
   - 跟进方式
   - 跟进备注
   - 操作人
51. 跟进方式至少支持：
   - 电话
   - 微信
   - 短信
   - 其他人工沟通
52. 系统应支持录入下一次计划跟进时间，以方便运营安排后续动作。
53. 若同一患者对同一试验重复提交，线索详情页必须可查看历史提交记录。
54. 若同一患者对不同试验提交，后台应能通过患者账号关联查看其历史报名概况。
55. 线索详情页必须支持人工记录研究中心协同字段，至少包括：
   - 推荐中心
   - 中心联系人
   - 中心反馈状态
   - 中心反馈备注
56. V1 不建设研究中心独立登录页；所有中心协同信息都由平台运营人工录入。
57. V1 不要求内置电话外呼、短信发送、微信 CRM 集成。

### 7. FAQ 与患者教育内容管理

58. 系统必须支持 FAQ 列表页。
59. 系统必须支持新建、编辑、发布、下线 FAQ。
60. FAQ 至少应包含以下字段：
   - 标题
   - 摘要
   - 正文
   - 分类
   - 病种 / 适应症标签
   - 关联试验
   - 状态
61. FAQ 状态至少支持：
   - 草稿
   - 已发布
   - 已下线
62. FAQ 应支持在首页、试验详情页、社区页按配置展示。
63. FAQ 编辑器在 V1 只需支持最小可用富文本能力，不要求复杂排版模板。
64. V1 不要求 FAQ 审批流和多版本协作。

### 8. 社区分区配置与社区审核

65. 系统必须支持按病种 / 适应症维护社区分区。
66. 社区分区配置至少应支持：
   - 分区名称
   - 分区简介
   - 是否开启
   - 推荐 FAQ
   - 推荐试验
67. 系统必须支持帖子与评论审核工作台。
68. 审核列表必须支持按内容类型、病种 / 适应症、审核状态、提交时间筛选。
69. 审核列表必须展示：
   - 内容类型
   - 标题或内容摘要
   - 作者
   - 是否匿名
   - 病种 / 适应症标签
   - 提交时间
   - 当前审核状态
   - 风险提示
70. 审核动作至少必须支持：
   - 通过
   - 驳回
   - 隐藏
   - 精选
71. 对帖子和评论的所有审核动作必须保留原因和审计日志。
72. 系统必须支持基础敏感词拦截结果在审核页可见。
73. 敏感词风险至少覆盖：
   - 联系方式外露
   - 售药 / 代购
   - 明显违规医疗宣传
   - 招募无关垃圾内容
74. 对于精选帖子，后台必须支持挂载相关试验与预筛入口。
75. 社区审核模块应支持查看“社区 -> 预筛启动”的基础归因数据。
76. 若排期允许，V1 可支持运营以官方身份回复帖子；若未实现，可先用官方患者账号在前台承接。

### 9. 基础统计

77. 系统必须提供基础统计页。
78. 基础统计页必须支持以下默认时间范围：
   - 今日
   - 最近 7 日
   - 最近 30 日
79. 基础统计页至少必须展示：
   - 首页访问量
   - 试验详情浏览量
   - 预筛启动数
   - 预筛提交数
   - 注册患者数
   - 社区浏览量
   - 社区到预筛启动数
80. 基础统计页应支持按试验维度查看预筛启动与提交数据。
81. 基础统计页应支持按来源页面维度查看数据，至少区分：
   - 首页
   - 试验详情
   - FAQ
   - 社区
82. V1 不要求自定义报表、漏斗自由配置和高级归因模型。

### 10. 系统配置与后台账号

83. `admin` 必须可以管理后台账号。
84. 后台账号管理至少支持：
   - 新建账号
   - 停用账号
   - 重置密码或重发初始化登录方式
   - 角色设置
85. 系统配置至少应支持维护以下基础字典：
   - 病种 / 适应症标签
   - 城市列表
   - 社区分区排序
   - 敏感词列表
   - 线索关闭原因
86. 病种 / 适应症标签体系必须兼容传统病种与医美相关适应症。
87. 医美相关适应症示例应至少覆盖：
   - 痤疮
   - 黄褐斑
   - 瘢痕修复
   - 脱发 / 雄激素性脱发
   - 色素沉着相关适应症
   - 皮肤年轻化 / 光电治疗相关适应症
88. V1 不要求配置中心支持多环境发布和复杂版本管理。

### 11. 审计与日志

89. 系统必须对以下关键动作保留审计日志：
   - 后台登录
   - 试验创建 / 编辑 / 上下线
   - 预筛表单发布
   - 线索状态变更
   - 跟进记录新增
   - FAQ 发布 / 下线
   - 社区审核动作
   - 系统配置修改
90. 审计日志至少应包含：
   - 操作人
   - 操作时间
   - 操作对象
   - 操作类型
   - 变更摘要
91. V1 不要求独立的高级审计中心页面，但相关日志数据必须可查询。

## Non-Functional Requirements

1. 性能：后台列表页在常见筛选条件下，P95 响应时间应小于 2 秒。
2. 性能：线索详情页和试验编辑页的 P95 打开时间应小于 3 秒。
3. 稳定性：后台月度可用性目标不低于 99.5%。
4. 安全：患者手机号等敏感字段必须在列表页脱敏展示，详情页仅对已登录后台用户可见。
5. 安全：后台必须具备基础会话校验和权限校验。
6. 合规：社区审核、FAQ 管理和试验文案必须可追溯到具体操作人。
7. 可维护性：后台必须保持单一 Web 管理台、单一数据库、固定权限模型，避免 V1 架构膨胀。
8. 可扩展性：数据模型应支持未来新增导出、自动提醒、中心协同增强等能力，但 V1 不提前实现。

## QA & Testing Impact

### High-risk surfaces

1. 试验公开状态与前台可见性的同步。
2. 项目级预筛表单配置与前台表单渲染的一致性。
3. 线索状态、跟进记录、我的报名页之间的数据一致性。
4. 社区审核状态与前台帖子、评论展示的一致性。
5. 试验详情页、社区帖子页中的预筛入口与后台配置的一致性。

### New test coverage required

1. `operator` 与 `admin` 的基础权限边界。
2. 试验从草稿到发布、从招募中到下线的完整流程。
3. 表单未发布时的风险提示与前台拦截逻辑。
4. 线索重复提交、重复患者、多项目报名场景。
5. 跟进记录新增后在线索详情和列表中的展示。
6. FAQ 发布 / 下线后的前台展示变化。
7. 社区帖子和评论的通过、驳回、隐藏、精选分支。
8. 病种 / 适应症标签同时覆盖传统病种与医美适应症的筛选与展示。

### Regression scope

1. 患者端试验列表、试验详情、预筛表单、我的报名。
2. 患者社区首页、病种社区、帖子详情、评论区。
3. FAQ 列表与 FAQ 详情。
4. 公开站首页推荐试验与推荐内容。

### Edge cases to verify

1. 试验已发布但预筛表单被下线后，前台 CTA 如何处理。
2. 线索负责人被停用后，历史线索如何展示。
3. 社区帖子被精选后再被驳回或隐藏时，前台推荐位是否同步下线。
4. 同一患者短时间内重复提交相同项目时，线索和报名记录如何保留。
5. 医美相关适应症项目与传统病种项目共用标签体系时，筛选是否正确。

## Success Metrics

由于模块尚未上线，基线默认视为 0。建议上线后前 90 天重点观察以下指标：

### 效率指标

1. 新建试验到公开发布的中位耗时低于 30 分钟。
2. 招募中试验里，已发布预筛表单的覆盖率达到 95% 以上。
3. 新线索首次被处理的 24 小时达成率达到 80% 以上。
4. 社区待审核内容的 24 小时处理完成率达到 90% 以上。

### 质量指标

1. 线索详情页中带有完整跟进记录的“已联系/跟进中”线索占比达到 95% 以上。
2. 因后台配置错误导致的前台无效预筛入口占比低于 1%。
3. 社区审核误放出的高风险内容数量维持在可接受范围内，首月目标为 0 起严重事故。

### 业务支撑指标

1. 通过后台发布的试验项目能稳定支撑平台级 `Weekly Qualified Prescreen Submissions` 增长。
2. 社区来源的预筛启动量可以被稳定归因和追踪。

## V1 实现边界建议

1. 后台采用单一 Web 管理台，不拆第二套研究中心系统。
2. 权限只做 `operator/admin` 两档，不做复杂 RBAC。
3. 试验管理、表单配置、线索池、FAQ、社区审核共用一套后台导航和一套账号体系。
4. 表单配置复用项目级预筛表单 PRD，不另起一套通用搭建器。
5. 线索处理先以“列表 + 详情 + 状态 + 跟进记录”完成闭环，不做自动化工作流。
6. 中心协同先通过线索详情里的人工字段承接，不做中心账号和中心回传门户。
7. 统计页只做固定指标卡和少量趋势图，不做分析平台。
8. 审计日志先做到“有数据可追溯”，不要求首版有复杂检索界面。
